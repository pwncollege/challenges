## Candidate 1: CVE-2025-12428 + b/452605803

Components: Wasm custom descriptors / JS-Wasm prototype chains plus Wasm import dispatch table exposure.

Relevant IDs: CVE-2025-12428, b/447613211 as first stage; b/452605803 as second stage.

First-stage fix: main `55496daf90227fb93311c535922f4b2142eeb72c` on 2025-10-06, "[wasm-custom-desc] Fix prototype validity cells".

Second-stage fix: main `7a322570d2973af0abf7f3122eb0674334d3af0a` on 2025-10-30, reverted, then relanded as `58bdae122c0f44e427ee2ec7dda636620e7aade2` on 2025-10-31.

Overlap window as inferred: from the unknown introduction of CVE-2025-12428 until 2025-10-06, while b/452605803 was still unfixed. The actionable mainline overlap ends at the first-stage fix on 2025-10-06.

Capability path: Wasm custom-descriptor type confusion or prototype-validity-cell bug gives in-cage object/type confusion (`HS-RW` potential per index) -> corrupt Wasm objects reachable from the cage -> swap or steer `WasmTrustedInstanceData::dispatch_table_for_imports` while it is still exposed -> use unexpected dispatch-table mutations to cross the heap-sandbox boundary.

Why it might work: both bugs are Wasm-adjacent. The second-stage fix explicitly defends against swapping attacks after in-sandbox corruption, which is the exact condition a Wasm first-stage bug might supply.

Blockers/uncertainties: the first-stage primitive is not proven as arbitrary cage R/W; it may require very specific custom-descriptor/prototype state. The second-stage public regression still needs review for helper-free reachability and for whether attacker-controlled mutation yields a native control primitive or only a crash.

Priority: High.
