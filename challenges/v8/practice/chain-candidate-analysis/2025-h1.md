## Candidate 1: CVE-2025-5419 + b/422645418

Priority: High.

Components:

- First stage: CVE-2025-5419, b/420636529, V8 Turboshaft stale JIT load / store-store alias-analysis issue.
- Second stage: Chromium issue b/422645418, Wasm JSPI suspender/native-stack corruption escape.

Relevant fixes:

- First-stage fix: `7bc0a67ebfbf44e7adab47fc2bbbe308660e27f4`, 2025-05-27 12:13:45 -0700, `[turbofan] Weaken alias analysis in store-store elimination`; the changed reducer file is under `src/compiler/turboshaft/`.
- Second-stage short-term fix: `25e22b601c7ddb0a3491cdfb5c604e9e589278d2`, 2025-06-05 04:34:45 -0700, `[wasm][jspi] Reintroduce an SBXCHECK`.
- Second-stage hardening: `9dcbe0576a48866e9f2c1a242fe6f875c55a6373`, 2025-06-12 07:46:59 -0700, `[wasm][jspi] Move suspender object to trusted space`.

Overlap window:

- Best inferred: from the unknown introduction of CVE-2025-5419 until `7bc0a67e` on 2025-05-27 12:13:45 -0700.
- Existing local chain revision: `5c198837c21b9b6cde113c4cb35d00e6b368f9a5`, dated 2025-05-27 10:01:01 -0700, before both the CVE-2025-5419 fix and the b/422645418 fixes.

Capability path:

- CVE-2025-5419 gives in-heap-sandbox object corruption / read-write primitives.
- The b/422645418 path corrupts JSPI `WasmSuspenderObject` state in the sandbox, specifically parent/stack relationships, to redirect a stack switch.
- The existing exploit material then uses Wasm stack contents as a native-stack ROP surface and calls the local challenge `catflag` helper.

Why it might work:

- The b/422645418 fix message is explicit: suspending was unsafe because it used the in-sandbox `WasmSuspenderObject` chain, which could be corrupted.
- The later hardening moves the suspender object to trusted space and protects sensitive fields, matching the capability needed for a sandbox escape.
- The repository already contains `chain-cve-2025-5419-422645418` with a private exploit that builds a V8 heap R/W primitive, aliases suspender stack fields, sprays stack words, and pivots into a ROP sequence.

Blockers / uncertainties:

- The true CVE-2025-5419 introducing commit is not identified in the index.
- The existing exploit uses revision-specific offsets and addresses.
  A new challenge variant would need offset regeneration or a more robust discovery phase.
- This is d8/challenge-oriented evidence, not a browser renderer sandbox escape.

## Candidate 2: CVE-2025-5959 + b/422645418

Priority: Hold.

Components:

- First stage: CVE-2025-5959, b/422313191, Wasm canonical equality type confusion.
- Second stage: b/422645418, Wasm JSPI suspender/native-stack corruption escape.

Relevant fixes:

- First-stage fix: `a970eed8995bb1b2cd083c3c79cdefc79e84f8b7`, 2025-06-05 07:09:22 -0700, `[wasm] Fix CanonicalEquality::EqualValueType`.
- Second-stage short-term fix: `25e22b601c7ddb0a3491cdfb5c604e9e589278d2`, 2025-06-05 04:34:45 -0700.
- Second-stage hardening: `9dcbe0576a48866e9f2c1a242fe6f875c55a6373`, 2025-06-12 07:46:59 -0700.

Overlap window:

- Best inferred fully-unpatched overlap: from the unknown introduction of CVE-2025-5959 until `25e22b60` on 2025-06-05 04:34:45 -0700.
- Existing local chain revision: `d7d46a7beb311a718780e77277f17d2edc9359df`, dated 2025-06-05 03:47:18 -0700, about 47 minutes before the b/422645418 short-term fix and about 2 hours before the CVE-2025-5959 fix.
- A speculative post-`25e22b60`, pre-`9dcbe057` window exists until 2025-06-12 only if the SBXCHECK can be bypassed or is insufficient for a specific corruption pattern.
  I did not find evidence for that in the local material.

Capability path:

- CVE-2025-5959 is indexed as HS-RW via Wasm type confusion.
- b/422645418 provides the same JSPI suspender/native-stack second stage as Candidate 1.

Why it might work:

- Existing `chain-cve-2025-5959-422645418` challenge material pairs these exact components at a revision before both fixes.
- The second-stage fix and hardening messages line up with the exploit's strategy of corrupting in-sandbox suspender fields.

Runtime status:

- On June 9, 2026, `pwnshop test challenges/v8/practice/chain-cve-2025-5959-422645418 --jobs 1 --timeout 300 --attempts 3` failed to print the flag.
- A direct run reached the canonical-equality collision, fake Wasm array, JSPI hook sequence, and final forged-stack handoff, then d8 segfaulted.
- Treat the first-stage and JSPI path as real source evidence, but do not treat the chain as a passing capstone until the final handoff is repaired.

Blockers / uncertainties:

- The overlap is narrow on mainline, and exact branch state matters.
- The CVE-2025-5959 exploit is Wasm-specific and likely more setup-heavy than the 5419 first stage.
- As with Candidate 1, offsets and native addresses are revision/build specific.

## Candidate 3: First Stage + Wasm / JS Dispatch Table Issues

Priority: Low-Medium.

Components:

- Candidate first stages: CVE-2025-5280, b/417169470 (`96d88f5f` branch fix on 2025-05-22; main fix `37d6fa3f` on 2025-05-19), CVE-2025-5419 if it existed before early May, or other HS-RW bugs whose vulnerable range predates late April / early May.
- Second-stage candidates: b/410295354 and b/413385716.

Relevant fixes:

- CVE-2025-5280 fix: `96d88f5f25963997c603e2fe0ec7c971e1af5245`, 2025-05-22 02:04:32 -0700, `[turboshaft] Fix map-based alias analysis in Late Load Elimination` on branch; mainline source fix `37d6fa3f39e17bf46d1cdf340e666cbd3ff976b3`, 2025-05-19.
- Dispatch-table fix 1: `53c5a5551a87056c7fe585131a28ef6bdc471add`, 2025-04-23 10:10:35 -0700, `[wasm] Use stricter types for dispatch table updates`.
- Dispatch-table fix 2: `dd6fe46039e902573fdbbce1af929367a54225ea`, 2025-05-05 11:34:31 -0700, `Load JS dispatch table from isolate`.

Overlap window:

- Best inferred with CVE-2025-5280: unknown first-stage introduction until 2025-04-23 for b/410295354, or until 2025-05-05 for b/413385716.
- There is no proof from the current index that CVE-2025-5280 was introduced before either dispatch-table fix; this is a candidate for commit-range triage, not a ready chain.

Capability path:

- First stage provides in-sandbox object / Wasm corruption.
- b/410295354 may allow overly permissive implicit-argument types during Wasm dispatch-table updates.
- b/413385716 states that process-wide cached wasm wrappers referenced a per-isolate-group / per-sandbox JS dispatch table, suggesting a cross-sandbox or wrong-dispatch-table boundary issue.

Why it might work:

- Dispatch tables and code pointers are trusted-memory / control-transfer surfaces.
- The b/413385716 commit explicitly mentions sandbox independence of generated wrappers and per-isolate-group dispatch tables.

Blockers / uncertainties:

- The fix subjects are hardening/typing and cache-correctness signals, not direct exploit claims.
- b/413385716 may require multi-isolate-group or multi-sandbox conditions not present in a standard d8 challenge.
- Need exact regression tests or a crafted corruption target before assigning higher priority.

## Candidate 4: Late-June First Stage + b/40070746 / b/435630461 TypedArray Switcheroo Window

Priority: Medium-Low.

Components:

- Candidate first stage A: CVE-2025-6554, b/427663123, optional-chain hole-check elision.
- Candidate first stage B: non-CVE b/428112179, RAB/GSAB deserialization of OOB TypedArrays.
- Second stage: b/40070746 / b/435630461, TypedArray ElementsKind switcheroo / native OOB access family.

Relevant fixes:

- First-stage A fix: `069790710f28b00ff8d7b4c665eef6b4eb8768f6`, 2025-06-26 10:48:31 -0700, `[interpreter] don't elide hole checks across optional chain`.
- First-stage B fix: `dba25156b08e8fbd76992fe5c1a2739e80a840c4`, 2025-06-30 06:08:39 -0700, `[rab/gsab] Fix deserialization of OOB TAs`.
- Related mitigation: `0efc70060ab7f22cb76cc1f0620df8a6123f9491`, 2024-11-28, increased trailing guard regions as a temporary workaround for b/40070746.
- Proper TypedArray-length fix: `8a56f0f07c9eedaf7cbbb6954a611cc0a3c194f9`, 2025-05-30, `[sandbox] Try 2: remove the TypedArray "length" field`.
- Guard-region reduction / practical window start: `b40a364440cbe4353a5eb01f567448872f5a1c6d`, 2025-06-24 06:35:30 -0700, `Revert temporary fix for issue 40070746`.
- Guard restored: `ba28598bb8ccb3fbefd2ffa0baaeaddb8b66be89`, 2025-08-05, reverting `b40a3644` due to b/435630461.
- TypedArray access checks: `fb9c018080153be9e7517c59913a314842cb0f4b`, 2025-09-16, and `2daccd5242836cd0f2fedbbdf4ea8ad450e0092b`, 2025-09-19.

Overlap window:

- CVE-2025-6554 path: 2025-06-24 06:35:30 -0700 through 2025-06-26 10:48:31 -0700, assuming CVE-2025-6554 was already present when `b40a3644` reduced the guard.
- b/428112179 path: 2025-06-24 06:35:30 -0700 through 2025-06-30 06:08:39 -0700.

Capability path:

- First stage corrupts in-sandbox TypedArray / RAB / GSAB state.
- Second stage targets TypedArray element-size / ElementsKind mismatch so `base + index * element_size` can point outside the sandbox when guard regions are insufficient.
- The intended effect is an out-of-sandbox native OOB read/write via TypedArray access.

Why it might work:

- The September b/435630461 fix message is explicit: an ElementsKind switcheroo can point outside the sandbox, and the sandbox guard region is not big enough to save it.
- The August revert restored the larger guard because of b/435630461, indicating the June 24 guard reduction created a practical exposure window.
- b/428112179 is directly about OOB TypedArrays, making it thematically close to the second stage.

Blockers / uncertainties:

- The useful window inside H1 is only two to six days depending on first stage.
- The May 30 TypedArray-length removal may already close some b/40070746 paths; b/435630461 appears to be a related but not identical residual issue.
