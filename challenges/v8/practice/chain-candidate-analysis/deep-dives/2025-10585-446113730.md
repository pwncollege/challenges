# CVE-2025-10585 + b/446113730 Wasm Dispatch/Code-Pointer Deep Dive

## Verdict

**DROP for the default x64 pwn.college V8 challenge environment. HOLD only as
an ARM64-specific research lead.**

The overlap is real, and the `b/446113730` regression is strong second-stage
evidence. The problem is the first stage: `CVE-2025-10585 / b/445380761` is an
ARM64 compiler zero-extension issue, with no local public regression proving a
helper-free arbitrary caged read/write primitive. The second stage also starts
from `--sandbox-testing` helpers (`Sandbox.MemoryView`, `Sandbox.getAddressOf`,
`Sandbox.getObjectAt`), so a challenge still needs a concrete first-stage
replacement for those helpers.

## Commit Evidence

### First stage: CVE-2025-10585 / b/445380761

Fix commit:

- `ec6c184783824fa8f974013aae28a45c36c0112f`
- Author date: 2025-09-16 16:40:24 +0200
- Commit date: 2025-09-16 08:30:21 -0700
- Subject: `[compiler] Don't assume that upper 32-bit of Int32MulOvfCheck are 0`
- Bug line: `Fixed: 445380761`
- File changed: `src/compiler/backend/arm64/instruction-selector-arm64.cc`

The fix changes `InstructionSelector::ZeroExtendsWord32ToWord64NoPhis`.
Before the fix, every `Word32` `OverflowCheckedBinop` was treated as
zero-extending to 64 bits. After the fix, `SignedAdd` and `SignedSub` still
return true, but `SignedMul` returns false. The commit message explains why:
ARM64 lacks a flag-setting 32-bit multiply, so V8 uses a 64-bit multiply and
compares `result.X()` and `result.W()` for overflow; that sequence does not
guarantee zeroed upper 32 bits.

Evidence strength:

- The bug is real and architecture-specific.
- The local fix has no public regression test or exploit-shaped test file.
- The local index classifies it as `HS-RW / HS-EXEC potential`, not proven
  arbitrary caged R/W.
- A nearby ARM64 Maglev hardening commit,
  `4fce2e4c38d7e358c4cd6f947b4d080d215d6cab` on 2025-09-18,
  `[maglev] Zero-extend output of Int32Mul*`, adds explicit zero-extension
  after ARM64 `Smull`, changes `Int32MultiplyOverflownBits` from `Asr` to
  `Lsr`, and always moves `res.W()` for `Int32MultiplyWithOverflow`. This
  reinforces the ARM64 multiply upper-bits theme, but it is not the exact
  `Fixed: 445380761` commit.

### Second stage: b/446113730 code-pointer masking

Fix commit:

- `bf854e1d872f6815d3fe3547fe5a31294f29e04a`
- Author date: 2025-09-22 14:31:54 +0200
- Commit date: 2025-09-23 04:45:34 -0700
- Subject: `[sandbox] Properly mask wasm code pointers to avoid OOB`
- Bug line: `Bug: 446113730`
- Files changed:
  - `src/codegen/arm64/macro-assembler-arm64.cc`
  - `src/codegen/x64/macro-assembler-x64.cc`

The commit message is direct: invalid Wasm code pointers, including the `-1`
sentinel, could lead to OOB access of the `WasmCodePointerTable`. The fix masks
the target to `[0, WasmCodePointerTable::kMaxWasmCodePointers)` before indexing
the table.

Pre-fix evidence at `61f5b4c8a5bb1dce7f3690e80140ab04d83993d3`
(`ec6c1847^`, the best first-stage overlap target):

- x64 `MacroAssembler::CallWasmCodePointer` loaded
  `ExternalReference::wasm_code_pointer_table()`, then did `shll(target, 4)`
  and added it to the table base. The comment only justified the shift for
  valid code pointers.
- x64 `CallWasmCodePointerNoSignatureCheck` similarly did `shll(target, 4)`
  and called through `Operand(kScratchRegister, target, times_1, 0)`.
- ARM64 `ResolveWasmCodePointer` did `Add(target, scratch,
  Operand(target, LSL, 4))` before loading the signature hash.
- ARM64 `CallWasmCodePointerNoSignatureCheck` computed the entry address with
  `Operand(target, LSL, kEntrySizeLog2)` and loaded the call target.

Post-fix, both architectures first mask with
`WasmCodePointerTable::kMaxWasmCodePointers - 1`, then shift by the entry-size
log2. This is relevant to x64 as a second stage, but the paired first stage is
not x64.

### Second stage: b/446113730 stale dispatch table on grow

Fix commit:

- `1d13848287a6e226862b1e1a90b6ae747c8a2ba2`
- Author date: 2025-09-22 14:12:41 +0200
- Commit date: 2025-09-23 10:08:56 -0700
- Subject: `[wasm] Clear old dispatch table on grow`
- Bug line: `Bug: 446113730`
- Files changed:
  - `src/wasm/wasm-objects.cc`
  - `test/mjsunit/sandbox/regress/regress-446113730.js`

The commit message states the sandbox concern: all users should be updated to
the new table, but in-sandbox corruption might still create uses of the old
table, so old entries are cleared to make the next old-table call crash.

Pre-fix `WasmDispatchTable::Grow` allocated a new table, copied live entries,
moved `offheap_data()->wrappers_`, updated protected users, updated
`dispatch_table0` where relevant, and returned the new table. It did not clear
the old table entries after updating users.

The fix adds:

- A loop over `old_length`.
- `old_table->Clear(i, WasmDispatchTable::kNewEntry)`.
- A comment saying this avoids dangling uses via in-sandbox corruption.

## b/446113730 Regression Mechanics

`test/mjsunit/sandbox/regress/regress-446113730.js` is unusually explicit and
exploit-shaped:

- It requires `// Flags: --sandbox-testing`.
- It constructs a `DataView(new Sandbox.MemoryView(0, 0x100000000))`.
- It defines helper primitives from sandbox-testing:
  `Sandbox.getAddressOf`, `Sandbox.getObjectAt`, `caged_read*`, and
  `caged_write*`.
- It knows `WasmTableObject` offsets for `maximum_length` and
  `trusted_dispatch_table`.
- It sprays `WebAssembly.Table` objects to infer exposed-trusted-object handle
  stride.
- It instantiates a Wasm module with an imported function and an exported
  function that calls the import through `dispatch_table_for_imports()`.
- It creates a growable table `tt`, then overwrites
  `tt.trusted_dispatch_table` with a computed target handle.
- It calls `tt.grow(0x10)`, causing dispatch-table growth while the original
  dispatch table remains reachable through the corrupted path.
- The comments say this clears index 0 in the new `WasmDispatchTable` data,
  drops the `WasmImportWrapperHandle` refcount to 0, and frees the entry in the
  Wasm code pointer table while the original dispatch table is still alive.
- It reclaims the freed `WasmCPT` entry with a type-compatible Wasm function.
- It then crosses call metadata by using `WasmImportData` as if it were
  `WasmTrustedInstanceData`.
- The test notes a layout coincidence: `kMemory0StartOffset ==
  kWasmImportDataSigOffset`, making memory accesses target `wasm::CanonicalSig`.
- It uses a signature read/write gadget to copy `ll` signature reps over `sl`
  signature reps, then calls `write64(0x424242424242n,
  0x4343434343434343n)`.

This is strong second-stage evidence. It shows why both fixes matter together:
old dispatch-table use can free/reclaim a Wasm code pointer table entry, and
unmasked invalid Wasm code pointers can index the table incorrectly.

## Helper-Free Viability

Current helper-free viability is **not proven**.

The regression supplies the hard part with sandbox-testing:

- Arbitrary caged reads and writes.
- Object addresses.
- Object fabrication through `Sandbox.getObjectAt`.
- Direct overwrite of `WasmTableObject::trusted_dispatch_table`.

A real challenge exploit must replace those helpers with a first-stage
primitive. `CVE-2025-10585` only gives local evidence of an ARM64 compiler
upper-32-bit zero-extension bug. It does not locally provide:

- A standalone JS PoC.
- A public regression producing object corruption.
- A demonstrated `addrof`/`fakeobj`.
- A demonstrated arbitrary caged write to the `trusted_dispatch_table` slot.

The Wasm module builder dependency is less concerning: a challenge can inline a
minimal builder or precompile byte arrays. The blocking helper is
`Sandbox.MemoryView`, not `wasm-module-builder.js`.

## ARM64 Specificity

The first-stage fix is ARM64-specific in code and commit text:

- It changes only `src/compiler/backend/arm64/instruction-selector-arm64.cc`.
- The commit message says the issue exists because ARM64 has no flag-setting
  32-bit multiplication.
- The flawed assumption is specifically that ARM64 `Int32MulOvfCheck`
  zero-extends when it can leave stale upper 32 bits.

The b/446113730 code-pointer masking fix is cross-architecture for ARM64 and
x64, but that does not make the full pair portable. This pair's first stage
does not execute on x64.

## Portability To Default x64 Challenge Environment

Default pwn.college V8 challenge work should assume Linux x64 unless the
challenge intentionally changes architecture. Under that assumption, this pair
is a poor target:

- The first-stage CVE does not apply to x64.
- The second-stage code-pointer OOB did apply to x64 before `bf854e1d`, but it
  still needs a separate x64-compatible caged R/W primitive.
- The included `b/446113730` regression is not helper-free and will not run in a
  normal d8 challenge without `--sandbox-testing`.
- An x64 challenge based on this second stage should pair it with a different
  x64 or Wasm first stage, for example one of the H2 Wasm custom-descriptor
  candidates already identified in `2025-h2.md`.

## Final Recommendation

Use this report as a negative/parking decision:

- **Drop** `CVE-2025-10585 + b/446113730` for default x64 challenge generation.
- **Hold** only if an ARM64-specific challenge is acceptable and someone first
  demonstrates that `CVE-2025-10585` can produce the caged write needed to
  replace the sandbox-testing helpers.
- **Keep the second stage separately** as a useful `b/446113730` primitive
  target for a different, helper-free x64/Wasm first stage.
