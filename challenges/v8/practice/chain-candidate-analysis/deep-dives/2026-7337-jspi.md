## Maglev First-Stage Evidence

The CVE-2026-7337 fix is specific and credible. The bad CL made
`BuildCheckSmi` skip the runtime tag check whenever `TryGetInt32Constant(object)`
returned a Smi-range value. The fix explains why this is wrong for tagged
inputs: value equality to a Smi-range constant does not prove Smi tagging.

The new regression `test/mjsunit/maglev/regress/regress-500880819.js` shows the
primitive shape:

- Create a real `HeapNumber(5.0)` through a `Float64Array`.
- Train Maglev through `BuildCheckNumericalValue` so the HeapNumber is recorded
  as a checked-value alternative after `CheckFloat64SameValue`.
- Store that tagged HeapNumber into an object field previously holding Smis.
- The vulnerable Maglev graph elides `CheckSmi`, so the HeapNumber pointer flows
  into a kSmi field path and through `StoreTaggedFieldNoWriteBarrier`.

The second regression, `regress-501789186.js`, gives the same issue with script
context slots. Both tests are first-stage evidence for type confusion and missing
write-barrier behavior. They are not yet evidence of a complete exploit
primitive: there is no local addrof/fakeobj/caged-RW implementation for this CVE.

## JSPI Lifetime Reachability

b/485784597 is directly relevant to a sandbox escape. Its regression
`test/mjsunit/sandbox/wasm-jspi-uaf.js` corrupts `WasmResumeData.trusted_suspender`
with `Sandbox.MemoryView`, then resumes through a stale suspender after the stack
returns and is released from the stack pool. The fix splits JSPI return handling
into `return_jspi_stack`, clears `suspender->set_stack(isolate, nullptr)`, and
unpublishes the trusted suspender before retiring the stack.

b/501147587 is the exception-unwind variant. Its regression
`test/mjsunit/sandbox/regress-501147587.js` documents the useful native target:

- `StackMemory` is freed outside the sandbox after last-resort GC releases the
  stack pool.
- The stale suspender EPT entry still resolves to the freed `StackMemory *`.
- `SwitchStacks<Inactive,Suspended>` reads the freed jump buffer.
- The test comments record `sizeof(StackMemory) = 184`, `jmpbuf_.state = +68`,
  `sp = +24`, `fp = +32`, `pc = +40`, and `parent = +56`.
- On non-ASan builds, the intended route is a 184-byte native heap spray with
  `state = Suspended` and controlled jump-buffer fields.

This is strong second-stage reachability evidence. The missing piece is a
helper-free way to get the needed `WasmResumeData` and suspender handle, then
perform the field corruption using only the CVE-2026-7337 primitive.

## Helper-Free Viability

Status: **not proven**.

Good news:

- JSPI is installed by default at the checked 14.6/14.7-era revisions when V8 is
  not `wasm_jitless`; `WebAssembly.Suspending` and `WebAssembly.promising` do
  not require an experimental JSPI flag there.
- Maglev is enabled by default on x64 builds with `V8_ENABLE_MAGLEV`; forcing the
  regression path still normally needs `--maglev` plus either natural warmup or
  `%OptimizeMaglevOnNextCall` during triage.
- The JSPI surfaces themselves are ordinary WebAssembly JS APIs, not d8 helpers.

Bad news:

- Both local JSPI regressions use `--sandbox-testing` and
  `Sandbox.MemoryView`/`Sandbox.getAddressOf` to discover or overwrite internal
  fields.
- The b/501147587 regression obtains the resume callback and suspender handle via
  sandbox-testing object walks. A challenge exploit must reproduce this through
  a real caged read primitive.
- No local exploit shows that CVE-2026-7337 can produce caged arbitrary read,
  caged arbitrary write, addrof/fakeobj, or targeted field overwrite.
- Native heap spraying for the freed `StackMemory` is only described in the
  regression comments; there is no local helper-free allocator spray or ROP proof
  for this exact 2026 stack layout.
