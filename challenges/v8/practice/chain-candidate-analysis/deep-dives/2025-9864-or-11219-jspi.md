# CVE-2025-9864 or CVE-2025-11219 + Early-H2 JSPI

Status: **Hold, with CVE-2025-9864 + b/432289371 as the best variant**.

This candidate has real local overlap between early-H2 first-stage bugs and
JSPI/Wasm sandbox escape surfaces, but it is not yet as strong as the existing
`CVE-2025-5419 + b/422645418` or `CVE-2025-5959 + b/422645418` chains. The
second-stage evidence for `b/432289371` is concrete: a V8 sandbox regression
corrupts `WasmSuspendingObject.callable` and reaches a Wasm import/function
signature confusion. The missing piece is a helper-free first-stage exploit that
upgrades either CVE to a precise in-cage write.

### CVE-2025-11219 / b/439772737

Fix:

- [`4293cb47afcbb1575f51e1cbd63c7cf8f21f85ff`](https://chromium.googlesource.com/v8/v8.git/+/4293cb47afcbb1575f51e1cbd63c7cf8f21f85ff),
  2025-08-20 09:08:36 -0700, main@101957,
  `[wasm] WA.validate(): always copy the wire bytes`, `Fixed: 439772737`.

The commit message is direct: `WebAssembly.validate()` previously obtained a
raw `base::Vector<const uint8_t>` from the caller's ArrayBuffer, then evaluated
the compile-time options parameter. Options evaluation could detach the
ArrayBuffer. The fix always copies wire bytes before options evaluation.

Regression evidence:

- `test/mjsunit/regress/wasm/regress-439772737.js` uses a `Proxy` on the options
  object. Reading `options.builtins` calls `buffer.transfer()` and `gc()` before
  `WebAssembly.validate(buffer, options)` continues.
- The vulnerable control flow is present at July overlap revisions, including
  `7f1ab230...^` and `49e49f879a...^`: `GetFirstArgumentAsBytes` returns a raw
  vector, then `ArgumentToCompileOptions(info[1], ...)` can detach, then
  validation uses the stale vector for non-shared buffers.

This is real UAF evidence, but it is weaker for this chain than CVE-2025-9864:
the local regression proves stale wire-byte use, not a targeted object write.
To use it as a first stage, we still need a heap layout that turns the detached
buffer backing store reuse into a precise in-cage primitive.

### b/427946951: StackMemory Writes

Workaround / exposure:

- [`95f557007d5ecd239a21a5a4f828a1bbd4851c11`](https://chromium.googlesource.com/v8/v8.git/+/95f557007d5ecd239a21a5a4f828a1bbd4851c11),
  2025-07-04 06:49:45 -0700,
  `[sandbox] Make Wasm stack memory accessible to sandboxed code`,
  `Bug: 427946951, 350324877`.

The commit says two memory ranges are made accessible to sandboxed code:

- Wasm stack memory, because it is used after stack switching.
- `wasm::StackMemory`, explicitly called an unsafe workaround because sandboxed
  Wasm stack-switching builtins write into the embedded `JumpBuffer`.

Fix:

- [`7f1ab230da706d53c6db22b642dc666673b49e49`](https://chromium.googlesource.com/v8/v8.git/+/7f1ab230da706d53c6db22b642dc666673b49e49),
  2025-07-30 10:35:17 -0700,
  `[wasm][jspi][sandbox] Avoid writes to StackMemory from sandboxed code`,
  `Bug: 427946951,42202153`.

The fix moves suspension/resumption jump-buffer writes and central-stack switch
writes out of sandboxed code into privileged C calls. It also removes the
`StackMemoryDeleter` / page allocation path that registered `StackMemory` as
unsafe sandbox extension memory.

Important build caveat:

- The unsafe `StackMemory` object allocation is under
  `V8_ENABLE_SANDBOX_HARDWARE_SUPPORT`.
- The local pwn.college challenge args seen in existing chain levels set
  `v8_enable_sandbox = true` but do not set
  `v8_enable_sandbox_hardware_support = true` by default.
- Therefore this is a strong escape-surface indicator, but a conditional target
  for this repo unless the challenge deliberately enables hardware sandbox
  support.

### b/432289371 / b/435489665: WasmSuspendingObject

Initial fix:

- [`49e49f879a604b7f11ed32b0579231e8a139d623`](https://chromium.googlesource.com/v8/v8.git/+/49e49f879a604b7f11ed32b0579231e8a139d623),
  2025-07-31 03:56:08 -0700,
  `[wasm] Fix sandbox escape via manipulated WasmSuspendingObject`,
  `Bug: 432289371`.

The commit explains the primitive exactly: the `callable` stored in a
`WasmSuspendingObject` must not be a Wasm function. The constructor enforces
that, but sandbox corruption can change it later. If it is changed to a Wasm
function, import processing can treat the same import as both a suspending
import and an exported Wasm function import.

Regression evidence:

- `test/mjsunit/sandbox/regress-432289371.js` uses `--sandbox-testing` to emulate
  the missing first-stage primitive.
- It creates `var suspending = new WebAssembly.Suspending(foo)`.
- It writes at `addrOf(suspending) + 0xc`, replacing `WasmSuspendingObject`'s
  compressed `callable` field with `addrOf(write) + 1`, where `write` is a Wasm
  export with signature `(ref struct, i64) -> void`.
- A second module imports that corrupted suspending object, re-exports the
  internal function as `funcref`, obtains `write64`, and reaches a mismatched
  call shape. The test asserts a `TypeError` around a write attempt to
  `0x0000011111111111`.

Follow-up hardening:

- [`aedcea406cd3c0ce1e2ea5fd77b2c08edd912b34`](https://chromium.googlesource.com/v8/v8.git/+/aedcea406cd3c0ce1e2ea5fd77b2c08edd912b34),
  2025-08-01 07:17:58 -0700,
  `[wasm] Forbid recreating WasmInternalFunction for Wasm imports`,
  `Bug: 432289371`.
  This adds an `SBXCHECK` in `WasmTrustedInstanceData::GetOrCreateFuncRef`.
  Its regression again uses sandbox memory to shape the corrupted import state.
- [`9c89058f3c979a063f84e8f7e1d89ce638929c4d`](https://chromium.googlesource.com/v8/v8.git/+/9c89058f3c979a063f84e8f7e1d89ce638929c4d),
  2025-08-04 09:40:26 -0700,
  `[wasm][sandbox] Check sandbox-safety of WasmSuspendingObject`,
  `Fixed: 435489665,432289371`.
  This rejects the bad state after resolving the callable and consolidates the
  protection.

This is the best second-stage target in the set because the regression shows the
exact object field and the exact import-processing inconsistency to corrupt.

## JSPI Mechanics

`WebAssembly.Suspending` and `WebAssembly.promising` are installed through the
JSPI feature path in `src/wasm/wasm-js.cc`. At the July/August target revisions,
`jspi` is in `FOREACH_WASM_SHIPPED_FEATURE_FLAG` with default `true` in
`src/wasm/wasm-feature-flags.h` (`Shipped in v13.7`). The command-line flag name
still exists as `--experimental-wasm-jspi`, but the feature is enabled by
default unless explicitly disabled.

For `b/432289371`, the exploit mechanics are:

1. Create a normal `WasmSuspendingObject` around a JS callable.
2. Use an in-cage write to replace its `callable` field with a Wasm exported
   function. The regression's offset is `+0xc` from the object address for the
   compressed field.
3. Instantiate another Wasm module that imports that suspending object.
4. Force import resolution into an inconsistent state: suspending import plus
   Wasm-to-Wasm import.
5. Re-export or recreate a `funcref` from that state, creating a signature
   confusion around the imported function.

For `b/427946951`, the mechanics are different:

1. JSPI suspension/resumption and central-stack switching save and restore
   `JumpBuffer` fields: SP, FP, PC, stack state, and stack limit.
2. Before `7f1ab230...`, sandboxed builtins wrote those fields directly, which
   required `StackMemory` to be accessible to sandboxed code in the hardware
   sandbox prototype.
3. The fix moves those writes into privileged C calls and validates state
   transitions with `SBXCHECK`.

This is potentially an escape surface, but it is less immediately useful than
`b/432289371` for ordinary heap-sandbox challenge builds.

## Helper-Free Viability

`b/432289371` regression uses `--sandbox-testing`, `Sandbox.MemoryView`, and
`Sandbox.getAddressOf` only to stand in for the missing first-stage primitive.
Those helpers must not be present in a challenge. A valid chain must replace
them with:

- first-stage address discovery or object-layout inference,
- a caged write to `WasmSuspendingObject.callable`, and
- enough caged read/write to find the exported Wasm function object to install
  as the callable.

Feature flags:

- JSPI itself should not require a non-default feature flag at the target
  revisions; `jspi` is shipped and default-enabled. Passing
  `--experimental-wasm-jspi` is harmless and matches existing local JSPI chain
  runners, but it is not the key blocker.
- CVE-2025-9864 requires Maglev. A runner that uses `--no-maglev` would kill the
  first stage.
- The CVE-2025-9864 regression uses `%OptimizeMaglevOnNextCall` and therefore
  `--allow-natives-syntax`; that is regression scaffolding, not a proven
  helper-free exploit. A challenge would need natural Maglev optimization or
  runner tuning such as low optimization thresholds.
- CVE-2025-11219's regression uses `--expose-gc` to make the UAF deterministic.
  A real exploit would need memory pressure or a non-helper GC trigger strategy.
- `b/427946951` needs `v8_enable_sandbox_hardware_support = true` to reproduce
  the most direct `StackMemory` sandbox-extension memory issue. Existing local
  chain args generally set only `v8_enable_sandbox = true`.

## Bottom Line

Keep this candidate on hold, not as an immediate build target. The
`WasmSuspendingObject` second stage is real and unusually concrete, but neither
CVE currently has local helper-free evidence for the exact primitive needed to
replace `Sandbox.MemoryView` in the regression. If more time is spent here, put
it into CVE-2025-9864 at `1959fe...`: try to turn the Maglev Smi/Float64
confusion into a single compressed-pointer write to
`WasmSuspendingObject.callable`. If that fails, drop the candidate rather than
moving to the weaker `WebAssembly.validate()` UAF path.
