## First-Stage Evidence

### CVE-2026-7899 / b/505481948

The fix is a two-line source change plus a new regression. Before the fix, `WasmLoadEliminationAnalyzer::ProcessPhi` only assigned a replacement when a Phi's inputs resolved to the same base. If a later pass found that inputs were not identical after all, the old replacement could remain. The fix explicitly sets `replacements_[op_idx] = OpIndex::Invalid()` in the non-identical-input case.

At `488364a9d`, the vulnerable shape is present: `ProcessPhi` does not clear stale Phi replacements in the `same_inputs == false` path.

The regression `test/mjsunit/regress/wasm/regress-505481948.js` is strong first-stage evidence:

- It uses Wasm GC arrays and a holder struct.
- It creates a big array and two small arrays.
- In a loop, optimized code can treat a Phi as the earlier big array even after the holder field changes.
- It then writes index `3` with `expandedVictimLen = 0x40` through the confused array reference.
- Correct behavior is `kTrapArrayOutOfBounds`; the regression asserts that the optimized function traps.

This is more than a harmless miscompile. The test shape is an optimized Wasm out-of-bounds write against a Wasm GC array, and the CVE index classifies the issue as `HS-RW potential`.

### CVE-2026-9973 / b/509268941

The follow-up fix says multiple Phis can depend on each other, so all loop-Phi replacements must be cleared up front during loop revisit. It adds `MaybeReplacePhi`, a `phi_replacements_backups_` side table, and explicit backup/clear/recompute logic for loop Phi replacements.

At `488364a9d`, none of that dependent-Phi logic exists. At `bb38f891`, the first fix is present, but the dependent-Phi loop-revisit fix is still absent. That means `CVE-2026-9973` plausibly exists both at the selected pre-UAF-fix revision and in the later post-`bb38f891` interval.

The strongest regression is `test/mjsunit/regress/wasm/regress-509268941.js`:

- It creates `big` length 10 and `small` length 1 Wasm arrays.
- It stores through `cur[3]` in a loop.
- It updates `cur` through `holder.field`, then changes `holder.field` to `small`.
- The test comments state that optimized code incorrectly elided the bounds check in later iterations.
- Correct behavior is again `kTrapArrayOutOfBounds`, including after `%WasmTierUpFunction`.

This is a real OOB-write-shaped first-stage, but for this chain it is not the best primary variant. If used only in its later standalone window after `bb38f891`, it no longer overlaps the dispatch-table UAF.

## Dispatch-Table UAF Evidence

The `b/503422307` fix message is unusually direct:

```text
There was a sneaky way to turn concurrent in-sandbox mutation
into a stale WasmDispatchTable pointer.
```

The diff supports that description.

In `src/wasm/module-instantiate.cc`, `ProcessExports()` moves from `Build_Phase1` to `Build_Phase2`. The new code also includes an explicit comment near export freezing:

```text
A well-timed concurrent mutation attack might manage to achieve
execution of user code here; that's why ProcessExports() runs as
part of Build_Phase2().
```

Before the fix, `WasmTrustedInstanceData::New` allocated and stored the `exports_object` early in `WasmInstanceObject`, and `ProcessExports()` reused that object during phase 1. After the fix, `WasmInstanceObject::exports_object` becomes `JSObject|Undefined`, and the exports object is populated during phase 2.

In `src/wasm/wasm-objects.cc`, `WasmDispatchTable::Grow()` already called `old_table->Unpublish(isolate)`. The fix adds a second cleanup layer:

- Replace `old_table->protected_uses()` with the empty protected weak fixed array.
- Iterate the old length.
- Clear each old entry with `DispatchTableClear(*old_table, i, WasmDispatchTable::kNewEntry)`.

This is direct stale-table mitigation. The old table being merely unpublished was not enough if some trusted object still retained or reached a stale pointer.

There is no local JS regression test or challenge exploit for `b/503422307`. The only public-local evidence is the fix text, the source movement around instantiation/export freezing, and the old-dispatch-table clearing.

## Concurrency Requirements

This second stage should be treated as concurrency/reentrancy-dependent.

Reasons:

- The commit message names "concurrent in-sandbox mutation".
- The moved `ProcessExports()` code explicitly references "well-timed concurrent mutation" and possible execution of user code.
- The relevant mutation window is during Wasm instantiation and export object finalization, not a plain post-instantiation table operation.
- The stale pointer is tied to old dispatch-table growth/republishing state, so a trigger likely needs table growth or table initialization racing/reentering against instantiation paths.

Practical implication: a normal single-threaded heap-sandbox RW primitive is probably insufficient by itself. A helper-free exploit likely needs a carefully shaped Wasm import/export/property path that executes user JS at the export-freezing point, then mutates table/instance state while the instantiation code still has trusted handles to old dispatch-table state. Worker-based concurrency might help in an unrestricted browser setting, but d8 challenge helpers commonly remove `Worker`, and a challenge should assume no `Worker` unless it is intentionally exposed.

## Helper-Free Viability

Current status: **not proven helper-free**.

Positive signs:

- The first-stage regressions are normal `mjsunit` Wasm tests and do not require `Sandbox.MemoryView`.
- The core bugs are in production Wasm/Turboshaft and Wasm instantiation code.
- The selected revision is a normal mainline revision before all three fixes.

Concerns:

- The regression tests use `d8.file.execute("test/mjsunit/wasm/wasm-module-builder.js")`; a challenge exploit would need to inline or recreate the minimal Wasm module builder logic.
- `regress-509268941.js` uses `%WasmTierUpFunction` and `--allow-natives-syntax` to force optimization. A helper-free challenge would need natural tier-up or a permitted flag, and should avoid `%` intrinsics.
- The `CVE-2026-7899` load-elimination bug has now been remade into stable helper-free in-cage read/write on the selected target.
- There is no local proof that the dispatch-table stale pointer can be reached without sandbox-testing APIs, `d8` helpers, or `Worker`.

So the first stage is challenge-viable, but the pair is not challenge-ready and
was dropped from the kept challenge set.
