## First-Stage Primitive Evidence

The CVE-2026-7337 fix message is unusually explicit. The vulnerable
`BuildCheckSmi` path skipped a tag-bit `CheckSmi` whenever
`TryGetInt32Constant(object)` returned a Smi-range value. That is sound for
non-tagged representations, but not for tagged values: a tagged value can be
numerically equal to a Smi-range constant while actually being a `HeapNumber`.

The regression `test/mjsunit/maglev/regress/regress-500880819.js` demonstrates
the primitive shape:

- Create a real `HeapNumber(5.0)` through `Float64Array`.
- Train Maglev so `BuildCheckNumericalValue` records the HeapNumber as a
  checked-value alternative after a numeric equality check.
- Store the tagged HeapNumber into an object field that Maglev believes is a Smi
  field.
- The vulnerable graph elides `CheckSmi`, so a HeapNumber pointer reaches a
  kSmi-field store path via `StoreTaggedFieldNoWriteBarrier`.

The companion regression `regress-501789186.js` applies the same issue to script
context slots.

This is credible object corruption, but it is not yet a complete exploit
primitive. Local evidence does not include addrof, fakeobj, arbitrary caged read,
arbitrary caged write, or a targeted overwrite of
`WasmTableObject.trusted_dispatch_table`.

## b/498095290 Mechanics

`534f607e` states the boundary directly: load the dispatch table from
`WasmTrustedInstanceData` instead of from the untrusted `WasmTableObject`.

Before the fix, high-value paths called methods such as:

- `WasmTableObject::Set(isolate, table, entry_index, element)`
- `WasmTableObject::Grow(isolate, table, delta, value)`
- `WasmTableObject::Fill(isolate, table, start, value, count)`
- `WasmTableObject::UpdateDispatchTable(isolate, table, entry_index, ...)`
- `table_object->ClearDispatchTable(entry_index)`

Those methods internally loaded `table->trusted_dispatch_table(isolate)`.
After the fix, runtime and instantiation paths first load:

```cpp
trusted_instance_data->dispatch_table(table_index)
```

and pass that `WasmDispatchTable` handle explicitly into the table operation.

The most important changed areas are:

- `Runtime_WasmFunctionTableSet`, `Runtime_WasmTableGrow`, and
  `Runtime_WasmTableFill` now pass the trusted instance's dispatch table.
- `InstanceBuilder::SetTableInitialValues` and `InstanceBuilder::LoadTableSegments`
  now pass the dispatch table from `WasmTrustedInstanceData` when initializing
  function-table entries.
- `WasmTableObject::SetFunctionTableEntry` and all `UpdateDispatchTable`
  overloads no longer derive the dispatch table from the table object.

The commit message names the exploit-relevant bad case: during instantiation,
the old behavior could write a pointer to a partially initialized
`WasmTrustedInstanceData` into an existing `WasmDispatchTable` associated with
another table or instance.

The preparatory CL `aee5deda` also matters. It changes
`WasmTableObject.trusted_dispatch_table` from "not set for non-function tables"
to "empty wasm dispatch table for non-function tables", removes the old
`Union<Smi, WasmDispatchTable>` shape, and makes `WasmTrustedInstanceData`'s
`dispatch_table(i)` return a `WasmDispatchTable` rather than
`Union<Smi, WasmDispatchTable>`.

## Is This Merely Hardening?

No, but it is not yet proven as a standalone escape.

Reasons it is exploit-relevant:

- The CL says the old source was the "untrusted `WasmTableObject`" and the new
  source is `WasmTrustedInstanceData`.
- The concrete sink is a trusted dispatch table, not a validation-only metadata
  check.
- The named failure mode crosses object ownership: partially initialized trusted
  instance data can be written into a dispatch table belonging to another table
  or instance.
- Earlier local sandbox regressions for b/446113730 and b/452605803 show that
  corrupting `WasmTableObject.trusted_dispatch_table` is a known useful target
  class. Those tests use `Sandbox.MemoryView` to transplant a table's
  dispatch-table handle, call ordinary `WebAssembly.Table.prototype.grow`, and
  reach Wasm wrapper/code-pointer/signature confusion.

Reasons it is still not a proven b/498095290 exploit:

- The b/498095290 CLs add no dedicated regression test.
- The local b/452605803 regressions prove the dispatch-table field class, but
  they are not the same bug. They target import dispatch-table exposure and rely
  on `--sandbox-testing`.
- The `534f607e` mechanics focus on instantiation/runtime paths. JS API paths
  still legitimately start from a JS-created table object, so a b/498 exploit
  must corrupt or alias module-owned table state and then enter the right
  module-instantiation/table-initialization operation.

## Helper-Free Viability

Status: **not proven**.

What can be done without helpers:

- The Wasm surfaces are ordinary JS/Wasm APIs: module instantiation, active
  element segments, table initialization, `table.set`, `table.grow`, and
  `table.fill`.
- The Maglev trigger is ordinary JS after warmup. Triage uses
  `--allow-natives-syntax` and `%OptimizeMaglevOnNextCall`, but a challenge can
  often replace that with deterministic warmup plus `--maglev` if needed.

What still needs helpers in local evidence:

- The only complete dispatch-table transplant examples use
  `--sandbox-testing`, `Sandbox.MemoryView`, and `Sandbox.getAddressOf`.
- CVE-2026-7337 is not yet upgraded locally to caged address discovery or
  precise writes into a `WasmTableObject` trusted-pointer field.
- b/498095290 has no local PoC showing how to get a partially initialized
  `WasmTrustedInstanceData` written into a cross-instance dispatch table.

Minimum helper-free proof needed:

1. Turn CVE-2026-7337 into a caged read/write primitive or a targeted
   `WasmTableObject.trusted_dispatch_table` overwrite.
2. Build two Wasm instances/tables where one table object's untrusted
   dispatch-table handle can be made to refer to the other's dispatch table.
3. Trigger instantiation or element/table initialization on `b1410b3d` so the
   old path writes the wrong trusted instance data into that dispatch table.
4. Show the resulting mismatched dispatch entry can be used for more than a
   crash or `SBXCHECK`: ideally a wrapper/signature/call-target confusion
   similar in strength to the b/452605803 regressions.
