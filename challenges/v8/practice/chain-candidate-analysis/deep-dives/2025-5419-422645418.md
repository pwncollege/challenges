## First-Stage Evidence

The `7bc0a67...` fix changes `src/compiler/turboshaft/store-store-elimination-reducer-inl.h` so dynamically indexed heap loads invalidate all tracked fixed-offset stores:

- Before the fix, only fixed-offset heap loads called `MarkPotentiallyAliasingStoresAsObservable(load.base(), load.offset)`.
- After the fix, fixed-offset loads still do that, but dynamically indexed heap loads call `MarkAllStoresAsObservable()`.

That matches a stale load/store-store-elimination primitive: a later dynamic array load can alias earlier fixed-offset stores, but the old analysis did not account for that aliasing.

The private exploit exercises this shape directly:

- `vuln_leak16(key)` performs dynamic indexed loads `o[key + n]`, changes the backing object, then performs many fixed-index stores.
- `vuln_fake_obj(key)` performs a dynamic indexed load from an array holding `obj`, then performs a fixed-index store.
- The exploit warms both functions, leaks the `large_arr` elements address, forges a fake array, and validates `fake_obj`, `addr_of`, `v8_read64`, and `v8_write64`.
- It prints and checks `PRIMITIVE_OK` only after a real read/write round trip succeeds.

This is helper-free first-stage evidence in the local exploit. It does not use `Sandbox.MemoryView`, d8 `%` intrinsics, `d8.*`, `read`, `load`, `os`, or similar testing helpers.

## Second-Stage Evidence

The `b/422645418` fixes line up exactly with the exploit's target fields.

At `5c198837...`, `src/wasm/wasm-objects.tq` has:

```text
extern class WasmResumeData extends HeapObject {
  suspender: WasmSuspenderObject;
  on_resume: Smi;
}

extern class WasmSuspenderObject extends HeapObject {
  stack: ExternalPointer;
  parent: WasmSuspenderObject|Undefined;
  promise: JSPromise;
  resume: JSObject|Undefined;
  reject: JSObject|Undefined;
}
```

At the same revision, `Isolate::SwitchStacks` validates the immediate parent only with:

```text
DCHECK_EQ(from->jmpbuf()->parent, to);
```

The short-term fix `25e22b60...` changes that to `SBXCHECK_EQ`. Its commit message says suspending is unsafe because it uses the in-sandbox `WasmSuspenderObject` chain, which can be corrupted. The later hardening `9dcbe057...` moves `WasmSuspenderObject` to trusted space, changes `WasmResumeData#suspender` to a trusted pointer, and changes `WasmSuspenderObject#parent` to a protected pointer.

The private exploit follows that model:

- It enables JSPI through the challenge runner with `--experimental-wasm-jspi`.
- It builds a Wasm module using `WebAssembly.Suspending` and `WebAssembly.promising`.
- It obtains `WasmResumeData.suspender` from a pending promise via normal object-field reads supplied by the first-stage primitive.
- It follows the in-sandbox `parent` chain across nested suspenders.
- It writes `suspender3.stack = suspender1.stack`, forcing a return from `suspender3` to skip `suspender2` and `suspender1`.
- It then writes `suspender4.stack = suspender2.stack` and sprays native stack words to pivot into an `execve("/challenge/catflag", ...)` ROP sweep.

This is concrete second-stage evidence, not just a generic JSPI crash path. The exact fields corrupted by the exploit are the fields that the June hardening removes from attacker-writable heap-sandbox storage.

The challenge patch removes most d8 helpers from the global template, including `version`, `write`, `writeFile`, `read`, `readbuffer`, `readline`, `load`, `quit`, `Realm`, `Worker`, `os`, and `d8`. The runner does not pass `--allow-natives-syntax` or `--sandbox-testing`.

Caveat: the final payload deliberately invokes `/challenge/catflag` through native control flow. That is the challenge goal, not a V8 testing helper. The ROP chain contains hard-coded d8/gadget/stack values, so it is helper-free but not revision-portable.

## Private Exploit Assessment

Static assessment: the existing private exploit exercises both real bugs.

Reasoning:

- The exploit cannot reach the JSPI stage unless the CVE-2025-5419 primitive succeeds; it validates the fake object and read/write primitive before building the Wasm/JSPI stage.
- The JSPI stage uses the first-stage `setField` primitive to mutate `WasmSuspenderObject.stack` fields.
- The exploit's intended control-flow effect is the exact `b/422645418` condition: corrupted in-sandbox suspender state causes stack switching to skip the immediate parent relationship that was only `DCHECK`ed in release builds.
- With d8 helpers stripped and no sandbox-testing API, a flag match from the private test would imply the chain used the first-stage heap-sandbox read/write and the second-stage JSPI native-stack effect.
