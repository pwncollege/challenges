## First-Stage Evidence

### CVE-2025-12428 / b/447613211

Fix: `55496daf90227fb93311c535922f4b2142eeb72c`, `[wasm-custom-desc] Fix prototype validity cells`.

The fix changes prototype tracking to include Wasm objects in JS/Wasm interleaved prototype chains:

- `JSObject::MakePrototypesFast` no longer bails out by restarting at a Wasm object's prototype; it skips Wasm objects and continues tracking JS prototypes.
- prototype user registration and validity-cell lookup switch from JS-only helpers to `IsAnyObjectThatCanBeTrackedAsPrototype`.
- the regression `test/mjsunit/regress/wasm/regress-447613211.js` builds a custom-descriptor Wasm object whose descriptor stores a JS prototype, optimizes a JS property load through that chain, deletes `proto.c`, and expects the optimized load to invalidate.

This is helper-free and useful, but it is not yet a caged arbitrary read/write primitive. The observed capability is stale optimized property/prototype state. It may be possible to amplify via stale object/property assumptions, but there is no local proof that it can write the `WasmTableObject.trusted_dispatch_table` slot needed by b/452605803.

### CVE-2025-13226..13230 / grouped fix

Fix: `48c3551179299151b044b2c161566465497c0407`, `[wasm-custom-desc] Fix subtyping`.

The commit fixes several separate custom-descriptor exactness/subtyping mistakes:

- descriptors and described types must have matching declared subtyping.
- imported `ref.func` must return inexact function types.
- `ProcessBranchOnTarget` cannot use static reachability with custom descriptors.
- `JSToWasmObject` must honor exact expected types.
- `ref.get_desc` must not preserve exactness when the runtime object is a non-trivial subtype of the instruction immediate.

The strongest local evidence is for b/446122633 / CVE-2025-13227. Later regression `test/mjsunit/regress/wasm/regress-446122633.js` constructs two incompatible structs:

- source layout: `[externref, i32]`
- destination layout: `[i32, ref null $si]`

It then uses custom-descriptor branch analysis to make Turboshaft treat an actually reachable path as unreachable, elide a cast, and store the source object in a destination-typed global. The fixed engine traps with `kTrapIllegalCast`; the vulnerable engine is expected to accept the mismatched typed global. This is close to the standard Wasm-GC type-confusion shape needed for caged address leaks and writes.

Other variants:

- b/446113732 / CVE-2025-13226 is covered by the descriptor/described subtype-validation part of the grouped fix. The completed challenge defines `$small_desc <: $big_desc` while `$small` is not a subtype of `$big`, a module shape that the fixed validator rejects. In the vulnerable engine, `ref.cast_desc_eq $big` accepts a real `$small` object when supplied with its exact `$small_desc` descriptor because `$small_desc` is statically a subtype of `$big_desc`. The cast returns a `$big`-typed reference to the smaller allocation, and the `$big` OOB fields overlap adjacent one-field structs to become addrof plus caged read/write.
- b/446124893 / CVE-2025-13228 has `test/mjsunit/regress/wasm/regress-446124893.js`, where `ref.get_desc` of a subtype was typed too exactly and could satisfy `struct.new_desc`/`struct.new` for the supertype. The completed challenge uses that exactness bug to feed a subtype descriptor into supertype allocation, producing a supertype-sized object with a subtype custom map. The subtype's OOB fields overlap adjacent one-field structs and become addrof plus caged read/write.
- b/446113731 / CVE-2025-13229 is covered by the imported `ref.func` exactness part of the grouped fix. The completed challenge imports a function as a supertype signature while providing an exported function with an explicit subtype signature. Vulnerable `ref.func` gives the import an exact supertype reference; after casting through the subtype and back through `ref.cast null` to the exact supertype, `br_on_non_null` is reachable at runtime but bottomed out by `WasmGCTypeAnalyzer`. The following reachable-but-analyzer-unreachable path stores a `[externref, i32]` struct into an `[i32, ref null $si]` global, yielding addrof plus caged read/write.
- b/446124892 / CVE-2025-13230 is covered by added `exact-types.js` JS-to-Wasm exactness checks. The completed challenge uses that boundary directly: JS obtains a real exact subtype object, passes it to a Wasm function whose parameter is `ref exact $small`, and the vulnerable `JSToWasmObject` path accepts it. Inside Wasm, `ref.get_desc $small` then gives the actual subtype descriptor while the static type remains exact `$small_desc`; `struct.new $small` allocates a supertype-sized object with a subtype custom map. The subtype's OOB fields overlap adjacent one-field structs and become addrof plus caged read/write.

## b/452605803 Second Stage

Fix: `58bdae122c0f44e427ee2ec7dda636620e7aade2`, relanding `7a322570d2973af0abf7f3122eb0674334d3af0a`.

The commit message is explicit: `dispatch_table_for_imports` reused `WasmDispatchTable`, did not expect modifications after creation, and in-sandbox corruption could swap it to cause unexpected modifications. The fix creates a separate `WasmDispatchTableForImports : TrustedObject` and changes `WasmTrustedInstanceData::dispatch_table_for_imports` from `WasmDispatchTable` to `WasmDispatchTableForImports`. Normal tables remain `WasmDispatchTable : ExposedTrustedObject`.

The regression `test/mjsunit/sandbox/regress/regress-452605803.js` is strong second-stage evidence:

- It uses `--sandbox-testing` only to supply the missing first-stage caged memory view and object addresses.
- It overwrites a `WebAssembly.Table` object's `trusted_dispatch_table` trusted-pointer slot with a handle computed to target an imports dispatch table.
- It calls ordinary `WebAssembly.Table.prototype.grow`.
- It then reclaims a Wasm code-pointer-table entry with a type-compatible Wasm function, steers an imported call through the stale/mismatched dispatch data, forges signature interpretation, and ends with a 64-bit memory write attempt.
- Follow-up `913fe00799479b252d11f65e4fc0ed76a63a7faa` only relaxes the test because the final call may throw `"type incompatibility when transforming from/to JS"` instead of CHECK-failing, which confirms the regression still reaches the wrapper/signature boundary.

Helper-free assessment: b/452605803 itself is not helper-free. The public regression depends on `Sandbox.MemoryView` and `Sandbox.getAddressOf` to get caged read/write and stable exposed-trusted-object handle arithmetic. Once a first stage supplies caged addrof/read/write, the remaining trigger is ordinary JS/Wasm API usage. The exact `target_ofs = 0x7` in the regression should be treated as revision/layout-specific and rediscovered for any challenge revision.

## b/446113730 Second Stage Alternative

## Exploit Path Needed

For a full chain, the first stage must provide at least:

1. A caged address leak for a controllable `WebAssembly.Table` object.
2. A caged write to its `trusted_dispatch_table` trusted-pointer field.
3. Enough caged reads or heap shaping to identify the exposed-trusted-object handle for a target instance's `dispatch_table_for_imports`.

Then the second stage can:

1. Swap the table object's dispatch-table handle to the imports dispatch table.
2. Invoke table mutation (`grow`, `set`, or a variant-specific path) to mutate imports dispatch-table state that was assumed immutable.
3. Reclaim or redirect Wasm wrapper/code-pointer/signature state.
4. Convert the resulting mismatched Wasm call path into a native memory read/write or flag read.
