## First Stage A: CVE-2025-6554 Evidence

The fixing diff is narrow and interpreter-specific.

Before `22e9d962`, `BuildOptionalChain` created one `HoleCheckElisionScope` around the entire optional-chain expression. The fix moves that scope into `OptionalChainNullLabelScope`, with the added comment:

```text
Use the same scope for the entire optional chain, as links earlier in the
chain dominate later links, linearly.
```

Mechanically, this means the vulnerable code could elide hole checks too broadly across optional-chain control flow. The local index labels this as `HS-RW potential`, but the public V8 commit evidence by itself only proves a missing hole-check condition. I did not find a local regression test or exploit that turns this into a concrete arbitrary in-sandbox read/write primitive, and I did not find evidence that it can directly corrupt a TypedArray map, `external_pointer`, `base_pointer`, `byte_length`, or backing-store metadata.

Assessment:

- Proven: a real bug fixed in `bytecode-generator.cc`.
- Not proven locally: a first-stage primitive strong enough to set up b/435630461 without `--sandbox-testing`.
- Risk: optional-chain/TheHole bugs can be serious, but this report should not assume cage RW without a reproducer.

## First Stage B: b/428112179 Evidence

The fixing diff in `dba25156` is more directly TypedArray-shaped.

Before the fix, `Deserializer<Isolate>::PostProcessNewJSReceiver` handled off-heap `JSTypedArray` deserialization like this:

```text
void* start = backing_store ? backing_store->buffer_start() : nullptr;
if (!start) start = EmptyBackingStoreBuffer();
typed_array->SetOffHeapDataPtr(main_thread_isolate(), start,
                               typed_array->byte_offset());
```

`JSTypedArray::SetOffHeapDataPtr` computes:

```text
Address address = reinterpret_cast<Address>(base) + offset;
set_external_pointer(isolate, address);
```

So if the deserialized backing store is missing/null and the serialized `byte_offset` is non-zero, the old path could set the TypedArray data pointer to `EmptyBackingStoreBuffer() + byte_offset`. The fix changes the null-backing-store case to call:

```text
typed_array->SetOffHeapDataPtr(main_thread_isolate(),
                               EmptyBackingStoreBuffer(), 0);
```

Assessment:

- Proven: the bug directly concerns deserialization of an out-of-bounds/off-backing-store TypedArray data pointer.
- Not proven locally: attacker-controlled snapshot/deserialization input in the pwn.college d8 challenge model. Normal challenge JS does not deserialize attacker-provided heap snapshots.
- Fit with second stage: thematically good because it is already TypedArray state corruption, but the entry surface is poor unless the challenge deliberately exposes snapshot deserialization.

## TypedArray Second Stage Mechanics

The b/435630461 evidence is strong, but helper-assisted.

The initial September fix `bebbd2a5489dafdcacb42450fd0e414205f1fc65`, 2025-09-05 04:12:36 -0700, says:

```text
Because of potential ElementsKind switcheroo, we might be pointing
outside the sandbox when doing the data + index * element size math, and
the sandbox guard region is not big enough to save us.
```

That fix was reverted by `4987c534f02e16e1871c7e3aaa6f45eac07582ab`, 2025-09-09 01:59:21 -0700, because of `b/443847151`.

The reland `fb9c018080153be9e7517c59913a314842cb0f4b`, 2025-09-16 07:41:22 -0700, repeats the same rationale and adds `InsideSandbox(...)` checks around TypedArray stores/copies in `src/objects/elements.cc`.

The stronger follow-up `2daccd5242836cd0f2fedbbdf4ea8ad450e0092b`, 2025-09-19 06:03:26 -0700, explains the more robust condition:

```text
Do a more robust check that the element size * length stays within
our expected limits. This protects against the ElementsKind switcheroo
and also protects us against a potential problem where we first do
SBCHECK(InsideSandbox(base_ptr + ...)) and then read the base_ptr again.
```

The regression `test/mjsunit/sandbox/regress-435630461.js`, added by the September fix, demonstrates the vulnerable shape:

- Allocate `arrayBuffer = new ArrayBuffer(kMaxSafeBufferSizeForSandbox)`.
- Create `dest = new Uint8Array(arrayBuffer)`.
- Read the `Uint8Array` map and an empty `BigUint64Array` map through `Sandbox.MemoryView`.
- Call `dest.set(new Proxy(...), kMaxSafeBufferSizeForSandbox - 1)`.
- In the proxy `length` getter, switch `dest`'s map from `Uint8Array` to `BigUint64Array`.
- In later property gets, switch the map back and corrupt `dest.external_pointer` to a huge sandboxed pointer value.
- Return a `BigInt` value so the switched BigUint64 path performs an 8-byte write where the Uint8 length/offset arithmetic expected only a 1-byte endpoint.

This proves the second-stage effect: if an attacker can mutate the destination TypedArray's map and external pointer at the right time, `TypedArray.prototype.set` can compute a native data pointer outside the intended sandbox-safe region.

Important limitation: the regression uses `--sandbox-testing` and raw object-field corruption. It is not helper-free.

## Helper-Free Viability

Current status: **not proven helper-free**.

What would be needed:

- A first-stage primitive capable of corrupting a live TypedArray's map and either `external_pointer` or equivalent backing-store/data-pointer state at a controlled re-entrant point.
- Enough reliability to hit `TypedArray.prototype.set` or a related copied/fill/reverse path during the ElementsKind mismatch window.
- No use of `Sandbox.MemoryView`, `Sandbox.getAddressOf`, `Sandbox.corruptObjectField`, `--sandbox-testing`, d8 native syntax, or file/system helpers.

What the local evidence provides:

- b/435630461 regression: exact map/external-pointer corruption recipe, but only with `--sandbox-testing`.
- CVE-2025-6554 fix: no local primitive that reaches map/data-pointer corruption.
- b/428112179 fix: a plausible OOB TypedArray state during deserialization, but no normal JS challenge path to feed attacker-controlled snapshots.

That makes this weaker than the existing 2025 JSPI chains, where local private exploits already demonstrate helper-free first-stage cage RW and then corrupt second-stage fields.
