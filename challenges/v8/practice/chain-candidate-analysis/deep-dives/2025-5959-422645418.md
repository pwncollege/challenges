## First Stage Evidence: CVE-2025-5959

The index maps CVE-2025-5959 / b/422313191 to HS-RW, fixed by `a970eed8995bb1b2cd083c3c79cdefc79e84f8b7`, and notes this exact chain revision. The V8 fix is narrow and matches the bug description: `CanonicalEquality::EqualValueType` previously compared only relative canonical type indexes for indexed value types, and the fix adds a check that the non-index bits also match.

At `d7d46a7`, the vulnerable code path still has:

- `EqualValueType` checks whether both types have indexes.
- If indexed, it returns only `EqualTypeIndex(type1.ref_index(), type2.ref_index())`.
- It does not compare nullability/ref-kind/shared/exactness bits outside the index.

The local private exploit exercises this shape directly. It builds two recursive type groups with carefully chosen nullable/non-null indexed reference fields, extracts a mismatched field through the canonical equality confusion, and turns the result into a fake large `WasmArray`:

- Type collision setup and nullable/non-null field generation: `chain-cve-2025-5959-422645418/tests_private/exploit.js` around lines 2191-2239.
- Confused `fakearr` function returning a `ref null` array from the mismatched struct field: lines 2251-2263.
- Exported `small_caged_read` and `small_caged_write` over that fake array: lines 2272-2291.
- Derived caged read/write plus `addrof`/`fakeobj`: lines 2341-2376.

That is sufficient first-stage evidence for a helper-free in-cage RW primitive on the selected revision.

## Second Stage Evidence: b/422645418

The short fix commit message for `25e22b60` is explicit: returning was already safe because `StackMemory` lives outside the sandbox, but suspending remained unsafe because it used the in-sandbox `WasmSuspenderObject` chain, which could be corrupted. The code change upgrades the parent-stack relation in `Isolate::SwitchStacks` from `DCHECK_EQ(from->jmpbuf()->parent, to)` to `SBXCHECK_EQ(...)`.

At `d7d46a7`, that relation is still only a `DCHECK`, and the build has `dcheck_always_on = false`, so release challenge builds do not enforce it. Also at `d7d46a7`, `WasmResumeData` stores a normal `suspender: WasmSuspenderObject`, and `WasmSuspenderObject` is still a sandbox heap object with writable `stack` and `parent` fields. The later hardening `9dcbe057` moves the suspender to trusted space, makes `WasmResumeData#suspender` a trusted pointer, and makes `WasmSuspenderObject#parent` protected.

The local private exploit matches that primitive:

- It creates JSPI surfaces with `new WebAssembly.Suspending(...)` and `WebAssembly.promising(...)`: exploit lines 2468-2471.
- It walks `Promise -> resume handler -> SFI function data -> WasmResumeData -> suspender`: lines 2473-2485.
- It obtains nested suspenders through parent fields: lines 2511-2515.
- It corrupts `suspender3.stack = suspender1.stack`, skipping intermediate stack parents: line 2522.
- It later corrupts `suspender4.stack = suspender2.stack`: line 2551.
- It uses a Wasm i64-local stack spray as the native ROP surface: lines 2432-2455.

This is credible second-stage evidence and aligns tightly with both the short fix and the later trusted-space hardening.
