## Summary

The most plausible 2024 H2 alternatives are not generic pairings of every
`HS-RW potential` CVE with every sandbox hardening CL. The better leads cluster
around Wasm GC/canonical-type first stages paired with Wasm trusted-memory,
dispatch-table, or code-pointer-table boundary fixes:

1. Wasm canonical-type first stages (`CVE-2024-12053`, `CVE-2024-12381`,
   `CVE-2024-12692`) plus Wasm dispatch-table / code-pointer-table issues
   (`b/350628675`, `b/42204123`, `b/350292240`).
2. Wasm or Maglev first stages fixed before late October plus JSPI native-stack
   hardening (`b/356419168`, `b/356176643`, `b/42202153`,
   `b/372749557`).
3. Late-2024 first stages plus `BackingStore::buffer_start` hardening
   (`b/381999810`) are lower priority because the documented problem is API /
   embedder-facing and may not be reachable from ordinary challenge JavaScript.

Known `CVE-2024-6100` chains are good challenge references but their overlap
with the documented second stages ends before 2024 H2 under the current local
mapping.

## Candidate A: Wasm Canonical-Type Confusion + Wasm Dispatch Table

Priority: high.

Components:

- First stage: Wasm canonical type confusion producing in-heap-sandbox object
  corruption / cage read-write.
- Second stage: Wasm dispatch table / trusted `WasmImportData::call_origin`
  hardening.

Relevant bugs:

- First-stage options:
  - `CVE-2024-12053`, b/379009132: relative type indexes leaked from the type
    canonicalizer, causing type confusion in callers.
  - `CVE-2024-12381`, b/381696874: incorrect equality of canonical value
    types.
  - `CVE-2024-12692`, b/382291459: canonical struct type comparison missed
    mutability.
- Second stage:
  - b/350628675 and b/42204123: one `WasmDispatchTable` per
    `WasmTableObject`; `WasmImportData::call_origin` becomes a protected field
    storing either a `WasmInternalFunction` or a `WasmDispatchTable`.

Fix commits and dates:

- `CVE-2024-12053`:
  - Main fix: `20d9a7f760c018183c836283017a321638b66810`, 2024-11-19.
  - Indexed branch fix: `3fdedec45691a3ab005d62c3295436507e8d277a`,
    2024-11-27.
- `CVE-2024-12381`:
  - Main fix: `a4d354fa54ba3e6a0f061c6b959be408ead5db95`, 2024-12-02.
  - Indexed branch fix: `aad03217f482b90f34ae559ca3492295f56e648e`,
    2024-12-05.
- `CVE-2024-12692`:
  - Main fix: `3852cf8b5bceed6a76e4537fc0aa191f9ed672a3`, 2024-12-05.
  - Indexed branch fix: `92ed656a375cc0fa3346f00f7e0d9faf0df04db2`,
    2024-12-10.
- Dispatch-table fix: `74caf5449508c72236970e5d9e01b7212d609122`,
  2024-12-02.

Overlap window as inferred:

- With `CVE-2024-12053`: `unknown-intro..2024-11-19` on main, or
  `unknown-intro..2024-11-27` on affected branch lines, assuming the dispatch
  table issue already existed before the first-stage fix.
- With `CVE-2024-12381`: up to 2024-12-02 on main; branch overlap is narrow
  because the first-stage branch fix and dispatch-table main fix both land in
  the same week.
- With `CVE-2024-12692`: up to 2024-12-02 if the dispatch-table issue is the
  limiting fix, because the first-stage fixes land after that.

Capability path:

1. Use a Wasm canonical-type bug to obtain in-cage type confusion and then
   object-field read/write, similar in shape to the `CVE-2024-6100` primitive.
2. Use the cage primitive to corrupt Wasm table/import/function metadata that
   is trusted by the dispatch path.
3. Attempt to redirect a Wasm call target or confuse a dispatch entry enough to
   reach a native code pointer or an arbitrary raw memory primitive.

Why it might work:

- The first-stage and second-stage components are both Wasm object/model
  issues, reducing the amount of cross-component setup.
- The dispatch-table fix explicitly moves/changes the trust boundary for
  `WasmDispatchTable` and `WasmImportData::call_origin`, which is exactly the
  kind of trusted-memory edge a cage read/write first stage can try to abuse.
- Existing local `chain-cve-2024-6100-7024` demonstrates that a Wasm
  type-confusion cage primitive can be used to patch Wasm function data and
  escalate to raw memory effects on an older revision. This is not proof that
  the H2 dispatch-table issue is exploitable the same way, but it is strong
  evidence for the general capability path.

Blockers / uncertainties:

- Introduction dates for all three first stages and for the dispatch-table
  issue are not yet mapped. The overlap is therefore inferred from fix order.
- The dispatch-table CL may be proactive sandbox hardening rather than a
  standalone reachable escape.
- Need regression tests or a PoC showing controllable corruption of the exact
  table/import path on a revision before `74caf5449508`.
- The exact affected branch matrix matters; several CVE rows point at branch
  merges rather than first mainline fixes.

## Candidate B: Wasm Canonical-Type Confusion + Wasm Code Pointer Table

Priority: high to medium.

Components:

- First stage: same Wasm canonical-type bugs as Candidate A.
- Second stage: Wasm code pointer table signature hardening.

Relevant bugs:

- First-stage options: `CVE-2024-12053` b/379009132, `CVE-2024-12381`
  b/381696874, `CVE-2024-12692` b/382291459.
- Second stage: b/350292240.

Fix commits and dates:

- First-stage fixes: same as Candidate A.
- Second-stage fix: `58f407806ad0ea83d8174dd701ba4b84c3cca14f`,
  2024-12-12. Follow-ups include `08a2541bb31e...` on 2024-12-16 and
  `58283795f22...` on 2024-12-18 for ports / x64 cleanup.

Overlap window as inferred:

- With `CVE-2024-12053`: `unknown-intro..2024-11-19` on main, or
  `unknown-intro..2024-11-27` on affected branches.
- With `CVE-2024-12381`: `unknown-intro..2024-12-02` on main, or up to
  2024-12-05 on branch rows.
- With `CVE-2024-12692`: up to 2024-12-05 on main, or 2024-12-10 on branch
  rows; the CPT fix is later, so the first-stage fix is limiting.

Capability path:

1. Obtain cage object corruption via a Wasm canonical-type bug.
2. Corrupt a Wasm call target / function metadata path that eventually loads
   from the `WasmCodePointerTable`.
3. Abuse missing signature validation in CPT entries before `58f407806ad0` to
   call code with an incompatible signature or attacker-useful calling
   convention.

Why it might work:

- The CL states that every CPT entry now stores and validates a signature hash
  and that previous sandbox signature checks were replaced by CPT checks. That
  suggests the pre-fix trust model had a meaningful missing validation point.
- The existing `CVE-2024-6100` challenge shows that a first-stage Wasm cage
  write can pivot through Wasm call-target state; CPT signature checks are a
  nearby trust boundary.

Blockers / uncertainties:

- Missing signature validation is not automatically arbitrary native execution;
  a useful target/signature mismatch must be found.
- Need to confirm whether attacker-controlled in-cage corruption can reach the
  relevant CPT handle or only adjacent trusted objects.
- First-stage fixes mostly land before 2024-12-12, so the candidate depends on
  the CPT issue already existing during those earlier windows.

## Candidate C: H2 First Stage + JSPI Native-Stack State Corruption

Priority: medium.

Components:

- First stage: any H2 first-stage with reliable in-cage read/write.
- Second stage: JSPI stack-switching / native-stack state validation.

Relevant bugs:

- Stronger first-stage options before the JSPI fix:
  - `CVE-2024-7971`, b/360700873: Wasm spill-loop-inputs bug, fixed by
    `9d79b3bac9189a5b9782a78c360f2f8813959cd4` on 2024-08-22.
  - `CVE-2024-8194`, b/360533914: excessive canonical types / `ValueType`
    index handling, fixed by `79f3f1276efa17a6172a0923dd13436ad8337a86` on
    2024-08-20.
  - `CVE-2024-8638`, b/362539773: WasmJSFunction wrapper tier-up type issue,
    fixed by `309f157dd68a9af1490d5d820cc928f095ac9b93` on 2024-09-04.
  - `CVE-2024-8904`, b/365376497: JSPI + lazy deopt, fixed by
    `000866bcb5d2f8868dbcb1b7b4a0d92dec9a006f` on 2024-09-13.
  - `CVE-2024-10230`, b/371565065: wasm-to-js wrapper tier-up with
    cross-instance indexed signatures, fixed by
    `5fcbf3954eb9f7f8221f068b5324e5b6f04b5839` on 2024-10-10.
  - `CVE-2024-10231`, b/372269618: default externref/exnref reference type
    confusion, fixed by `d9893f4856af26e78ba5021063ee2b1c61a3023b` on
    2024-10-16.
- Second-stage options:
  - b/356419168, b/356176643, b/42202153:
    `8cccb138760e28d4c7ad8f89b6dd774f641f01f1` on 2024-10-17; reverted and
    relanded as `a9abc1649e1e5793af028a1b73992df651ceee5f` on 2024-10-17.
  - b/372749557: generator suspend/resume parameter-count hardening,
    `552c4d6ecbbc2d0513d5ea40a112de776836cadc` and
    `1d49cdf95498fe07ca867777f1a942eac2045bbd`, both 2024-10-29.

Overlap window as inferred:

- For the stack-state issue: `unknown-intro..min(first-stage fix, 2024-10-17)`.
  The most interesting late windows are `CVE-2024-10230` through 2024-10-10
  and `CVE-2024-10231` through 2024-10-16.
- For the suspend/resume parameter-count issue:
  `unknown-intro..min(first-stage fix, 2024-10-29)`. This includes first stages
  fixed before 2024-10-29 and may include `CVE-2024-11395` if that JSPI
  unwinder bug can be treated as a first-stage, but the index currently only
  marks it as `HS-RW potential`.

Capability path:

1. Use a first-stage in-cage primitive to corrupt JSPI stack/suspender/generator
   state reachable from Wasm or JS.
2. Trigger JSPI stack switching or generator resume so native stack metadata,
   parameter counts, or stack-state transitions are trusted.
3. Convert a bad stack transition or wrong parameter count into a native-stack
   write/read or controlled call frame.

Why it might work:

- The JSPI CL says it checks and updates target stack state to prevent stack
  corruptions under the sandbox model. That is a direct statement of a
  sandbox-relevant native-stack boundary.
- The generator fixes explicitly avoid trusting in-heap metadata such as
  `SharedFunctionInfo::formal_parameter_count` or dispatch-table parameter
  counts in suspend/resume paths.
- Several first-stage candidates are themselves Wasm/JSPI-adjacent, reducing
  setup distance.

Blockers / uncertainties:

- JSPI may require feature flags or challenge args depending on the target
  revision.
- Stack corruption does not imply an easy return-address overwrite; modern V8
  stack layouts and checks may make this crash-only.
- Need a concrete corrupted field and a repeatable trigger path. The current
  evidence is from hardening descriptions, not a public exploit.

  - `CVE-2024-12692`, b/382291459:
    `3852cf8b5bceed6a76e4537fc0aa191f9ed672a3` on 2024-12-05 main,
    `92ed656a375cc0fa3346f00f7e0d9faf0df04db2` on 2024-12-10 branch.
  - `CVE-2024-12693`, b/382190919:
    `e4ecfc909687511aeb20b88ce6ae2a7a1a80afe5` on 2024-12-05 main,
    `aad648bd2af9815d0c48eeb78cbf3d8e6471d094` on 2024-12-11 branch.
  - `CVE-2024-12695`, b/383647255:
    `357d0dd4bc7f64eb81cdf49c5cf3699cf151909d` on 2024-12-13 main,
    `cb0d9e1d7b8889192e417363d0d7280a2ea114fa` on 2024-12-16 branch.
- Second stage:
  - b/381999810, reland tracked with b/6074884:
    `2f38832cd8c088b05ce6642c473521af9b6deeb8` on 2024-12-05, reverted by
    `140ae5d27dd83aeeac4787b3fd41dea7e46485fc`, then relanded as
    `12a53cc493f02f64f30dd08c0ec03a866ae45481` on 2024-12-05.

Overlap window as inferred:

- With `CVE-2024-12053`: overlap ends at the first-stage fix, before the
  BackingStore fix lands.
- With `CVE-2024-12381`: overlap is narrow on main and may be branch-dependent.
- With `CVE-2024-12692`, `CVE-2024-12693`, and `CVE-2024-12695`: overlap can
  extend to the BackingStore reland on 2024-12-05 or to the first-stage fix,
  depending on branch.

Capability path:

1. Use an H2 first-stage to corrupt an `ArrayBuffer` / `TypedArray` backing
   store state or produce a malicious empty-buffer/offset combination.
2. Trigger an API path where `BackingStore::buffer_start()` returning `nullptr`
   leads an embedder to compute `nullptr + ByteOffset()`.
3. Attempt to turn that into an out-of-sandbox pointer in the first 32 GB of
   address space.

Why it might work:

- The fix text directly describes a sandbox boundary problem: ArrayBuffer
  buffer pointers must point into the sandbox, and `nullptr + ByteOffset()` can
  produce an address outside the intended cage model.

Blockers / uncertainties:

- This looks API / embedder-facing. Ordinary d8 JavaScript may not expose a
  path that performs the unsafe pointer arithmetic.
- The fix is designed to make bad accesses crash in a guard region; turning the
  pre-fix behavior into useful read/write likely requires a cooperating API
  consumer.
- Lower priority unless a challenge-specific embedder path is acceptable.

## Candidate E: Non-CVE Wasm Wrapper Overflow + Wasm CPT / Dispatch-Table Escape

Priority: medium-low.

Components:

- First stage: non-CVE Wasm wrapper overflow / in-cage corruption lead.
- Second stage: Wasm dispatch table or CPT trusted-memory issue.

Relevant bugs:

- First stage:
  - b/384844209: `[wasm] Fix buffer-overflow check in generic js-to-wasm
    wrapper`, `ae6c7c8f9e8336e1272a129d5d2acb83da4854f2`, 2024-12-23.
- Second-stage options:
  - b/350628675 / b/42204123: `74caf5449508c72236970e5d9e01b7212d609122`,
    2024-12-02.
  - b/350292240: `58f407806ad0ea83d8174dd701ba4b84c3cca14f`,
    2024-12-12.

Overlap window as inferred:

- `unknown-intro..2024-12-02` for the dispatch-table issue, or
  `unknown-intro..2024-12-12` for the CPT issue, assuming b/384844209 existed
  before those fixes. The first-stage fix on 2024-12-23 is not limiting.

Capability path:

1. Abuse the generic js-to-wasm wrapper buffer-overflow check to get a controlled
   in-cage overwrite or stack-adjacent corruption.
2. Pivot into Wasm dispatch/CPT metadata as in Candidates A/B.

Why it might work:

- The first and second stages are both in Wasm wrapper/call machinery, which is
  where dispatch tables and CPT entries are consumed.
- The late first-stage fix means it plausibly coexisted with both December
  sandbox-boundary fixes.

Blockers / uncertainties:

- `NON_CVE_RELEVANT_ISSUES.md` only labels this `HS-RW potential`; no local PoC
  or exact primitive was found.
- Need to inspect the regression test and diff to confirm whether this is
  reachable from JavaScript and whether the overflow is controllable.

- `CVE-2024-7024` challenge-label path: corrupt Wasm function data to get raw

Why it might work:

  exact chains.
- The b/344963941 fix explicitly prevents reading/popping an empty regexp
  backtrack stack.
- The public b/334120897 fixes explicitly address V8 sandbox escapes via Wasm

Blockers / uncertainties:

- These overlap windows end before 2024-07-01, so they are not H2 candidates
  under a strict temporal filter.
- The local `chain-cve-2024-6100-7024` label does not cleanly match the public
