## Candidate 1: CVE-2024-6100 + b/344963941 Irregexp Native-Stack Escape

Priority: High.

Evidence level: existing-test backed. The repo contains
`chain-cve-2024-6100-344963941/tests_private/exploit.js`, and
`chain-cve-2024-6100-344963941/challenge/REVISION` is
`1778ccde8eb86a01b4c16c34490a597235203b2d` from 2024-04-13.

Components:

- First stage: CVE-2024-6100, likely b/344608204.
- Second stage: Chromium issue b/344963941, also tied to b/42204606 in the
  fix commit.
- Subsystems: Wasm canonical type handling, RegExp interpreter bytecode and
  native backtrack stack.

Fixes and overlap:

- First-stage fix: `422cdc5eddcadb53b8eafb099722fb211a35739e`,
  2024-06-06, `[wasm] Enforce maximum number of canonicalized types`.
- Second-stage fix: `8859e5e21f4b5e587d9b75839c27befbdf1b9ddd`,
  2024-06-07, `[regexp] Harden backtrack strack access in interpreter`.
- Best inferred overlap window: at least 2024-04-13 through 2024-06-06, based
  on the working challenge revision and the first-stage fix date. The true
  overlap may be wider, but the introducing commits are not identified in the
  inventory.

Capability path:

1. CVE-2024-6100 creates a caged object-field read/write primitive from Wasm
   type confusion.
2. The exploit uses that primitive to locate and patch `JSRegExp` internals:
   the RegExp data object and its bytecode array.
3. The patched RegExp bytecode manipulates interpreter backtrack-stack state.
4. The second stage emits writes to native stack slots and builds a small ROP
   sequence ending in `execvp("/challenge/catflag", argv)`.

Why it might work:

- The existing exploit demonstrates that CVE-2024-6100 supplies enough caged
  write control to modify RegExp bytecode and metadata.
- The RegExp fix commit directly says it prevents reading/popping on an empty
  stack. The challenge uses precisely that class of underflow/stack-state
  behavior as the out-of-cage effect.
- The selected revision predates both fixes.

Blockers and uncertainties:

- The exploit is revision-specific: native stack addresses, `d8` base, gadget
  offsets, and RegExp object offsets are hard-coded in the test.
- The exact introduction date of b/344963941 is unknown, so overlap is proven
  by the challenge revision, not by a full affected-range analysis.
- This should be treated as a d8 challenge chain, not a browser renderer
  sandbox escape.

## Candidate 2: CVE-2024-6100 + CVE-2024-7024-Labelled Wasm Trusted Call-Target Escape

Priority: High for challenge work, with mapping caveat.

Evidence level: existing-test backed for the challenge behavior, but the public
CVE/bug mapping is incomplete. The repo contains
`chain-cve-2024-6100-7024/tests_private/test_exploit.sh`, and
`chain-cve-2024-6100-7024/challenge/REVISION` is the same 2024-04-13 V8
revision, `1778ccde8eb86a01b4c16c34490a597235203b2d`.

Components:

- First stage: CVE-2024-6100, likely b/344608204.
- Second stage: challenge-labelled CVE-2024-7024; best public-code fit is the
  b/336507783 / b/42204510 Wasm sandbox-hardening line around Wasm function
  data, dispatch/call-target state, and `WasmApiFunctionRef`.
- Subsystems: Wasm canonical types, Wasm exported function data,
  `WasmApiFunctionRef`, trusted pointer table, Wasm call targets.

Fixes and overlap:

- First-stage fix: `422cdc5eddcadb53b8eafb099722fb211a35739e`,
  2024-06-06.
- Second-stage best inferred closing fix:
  `388e59d87cd783625f2b8506359ffc9f648e4bbd`, 2024-07-01,
  `[wasm][sandbox] Un-expose WasmApiFunctionRef`.
- Related hardening that may partially constrain the same family:
  `e25efe6cf28b8a7dfc6de89ac16fd2a77f5e281d` on 2024-06-11
  verifies signatures in JS-to-Wasm wrappers;
  `f603d5769be54dcd24308ff9aec041486ea54f5b` on 2024-06-12 checks
  signatures when updating tables;
  `2b620f7de169ee3374ca1cb11098cc38ec2b0b8d` on 2024-06-27 checks
  signatures when importing tables.
- Best inferred overlap window for both working stages: at least 2024-04-13
  through 2024-06-06. The second-stage surface appears to persist beyond that
  until the June/July hardening sequence, but CVE-2024-6100 is fixed on
  2024-06-06.

Capability path:

1. CVE-2024-6100 gives caged reads/writes.
2. The exploit rewrites Wasm function data / function references so an exported
   Wasm memory read/write pair can operate as a raw memory primitive relative to
   the Wasm memory backing store.
3. It computes the cage base and `Isolate`, then reads the trusted pointer table.
4. It resolves a target Wasm function's trusted object from a function ref.
5. It overwrites the target internal function call target with a native gadget
   and calls the Wasm function to reach `execvp`.

Why it might work:

- The existing exploit validates the complete path on the April 2024 revision.
- The July 1 `WasmApiFunctionRef` change explicitly says the object is only
  referenced from trusted-space objects and should use protected pointers rather
  than the trusted pointer table. That lines up with the exploit's dependency
  on resolving a trusted object through a function-ref handle.
- The June signature-check commits explicitly describe preventing escapes from
  corrupted sandbox state by validating Wasm signatures and dispatch-table
  updates.

Blockers and uncertainties:

- The index states that the exact CVE-2024-7024 public mapping still needs
  confirmation. New work should not cite `388e59d...` as "the CVE-2024-7024
  fix" without confirming the CVE assignment.
- The existing exploit hard-codes offsets for `Isolate`, metadata tables,
  trusted pointer table, gadgets, and `d8` image layout.
- The second-stage boundary may be a hardening family rather than one discrete
  bug. Signature-check commits may close related paths before the July 1
  `WasmApiFunctionRef` change closes the specific exposed-object path.

Capability path:

1. Use CVE-2024-4947 as a caged object corruption or caged R/W first stage.
2. Recreate the RegExp metadata/bytecode patching strategy from Candidate 1.
3. Use b/344963941 to turn corrupted RegExp interpreter state into native-stack
   writes and a ROP pivot.

Why it might work:

- The index classifies CVE-2024-4947 as `HS-RW`, `HS-EXEC` potential.
- Candidate 1 shows that b/344963941 does not inherently depend on
  CVE-2024-6100; it needs a caged write primitive strong enough to patch RegExp
  internals.
- CVE-2024-4947 is close in time to the proven April 2024 chain revision, so
  object layouts may be more portable than for much older January bugs.

Blockers and uncertainties:

- The actual CVE-2024-4947 exploit primitive must be checked. If it is narrow,
  unstable, or only reaches a limited object set, it may not patch the required
  RegExp bytecode and length fields.
- The exact b/344963941 introduction date remains unknown.
- This candidate still needs a combined exploit and offset audit.

  primitive without a separate first stage.
- The index does not identify CVE-2024-2887's introducing commit, so overlap

- For challenge construction, this is better treated as a standalone
  CVE-2024-2887 challenge or as a baseline comparison, not as a priority chain.

## De-Prioritized H1 Leads

Several non-CVE H1 rows in `NON_CVE_RELEVANT_ISSUES.md` are relevant but should
not be promoted to chain candidates without more evidence:

- b/327473424, fixed by `76ae04522173630185977fb72f3fc8783a3a9911` on
  2024-02-29, is a native-stack OOB write after corrupted heap string length.
  It is interesting as an escape sink, but it needs proof that a JS-reachable
  first stage can drive the exact `Intl::ToICUUnicodeString` fast path with a
  corrupted length on a usable revision.
- b/329171423, fixed by `a0ee8d65eadb6a9a3ad391b7346db0f9f443adab` on
  2024-03-13, hardens the experimental RegExp engine against sandbox escapes.
  The commit also adds checks to ensure the experimental engine is enabled via
  runtime flag, so helper-free/default-d8 reachability is uncertain.
- b/325878101 and b/325893559 are Wasm dispatch-wrapper bounds/missing-table
  fixes from February 2024. They are chain-adjacent, but the current evidence
  points more strongly to the later b/336507783 / Wasm trusted-call-target
  hardening line for a practical escape sink.
- Many rows labelled `SBX-ESC potential / sandbox hardening` are defensive
  hardening for corrupted-heap robustness. They are useful for diff mining, but
  not enough by themselves to claim a plausible two-stage exploit chain.
