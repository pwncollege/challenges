## CVE-2024-4947 Primitive

Fix-side evidence:

- Before `b3c01ac1...`, `AccessorAccessInfoHelper()` returns
  `PropertyAccessInfo::ModuleExport(...)` for a `JS_MODULE_NAMESPACE_TYPE`
  holder without rejecting store/define access.
- The fix adds an `IsAnyStore(access_mode)` check and returns invalid access
  info for module namespace stores, matching the spec behavior that such stores
  are errors or no-ops.
- The fix also changes Maglev's property-store path so unsupported
  `PropertyAccessInfo` kinds, including `kModuleExport`, become unreachable
  instead of flowing into a data-field store path.

Primitive shape:

- The public write-up pins the same revision,
  `473318dfdb09464902c7633cad03b16431145800`.
- The bug produces a Maglev store where the receiver's properties pointer is
  loaded and a tagged value is written at field offset 0, effectively a write
  to the `PropertyArray` map slot.
- Public exploit development shows this can be turned into GC-driven
  `PropertyArray` shrinking and adjacent-object OOB access, then into object
  address disclosure and object/ArrayBuffer corruption.
- Securelist's in-the-wild analysis describes the same missing module-export
  store check, then a path that overwrites array and ArrayBuffer state to obtain
  memory read/write from JavaScript.

For this candidate, the important threshold is lower than a full process
read/write: the first stage needs caged writes to ordinary V8 heap objects so it
can patch a `JSRegExp` data array and `ByteArray`. The public primitive is
strong enough in principle because it reaches JS object/property/array metadata
inside the compressed-pointer cage. However, the current local
`cve-2024-4947` challenge is not a proof of the sandboxed version: it builds
with `v8_enable_sandbox = false` and exposes `%` natives plus `gc()`.

Conclusion: **enough in principle, not locally proven in the final harness**.
