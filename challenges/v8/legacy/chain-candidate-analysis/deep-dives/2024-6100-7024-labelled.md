# CVE-2024-6100 + CVE-2024-7024-Labelled Wasm Path

Generated: 2026-06-03.

Verdict: **Keep**, with mapping and window caveats.

This is a real local chain candidate, not a helper-assisted sandbox-testing
demo. The checked-in exploit uses a CVE-2024-6100 Wasm canonical-type
first-stage primitive to corrupt ordinary in-cage Wasm function metadata, turns
that into raw virtual-address read/write through the public CVE-2024-7024 /
b/334120897 Wasm path, then uses the resulting raw write to patch a trusted
Wasm internal-function call target and execute `/challenge/catflag`.

The caveat is naming precision: the local challenge label should be read as
`CVE-2024-7024 / b/334120897, plus adjacent Wasm trusted-function hardening`,
not as a clean one-commit `CVE-2024-7024 == 388e59d...` mapping.

## First-Stage Primitive Evidence

The local exploit builds a helper-free CVE-2024-6100 primitive:

- Lines 56-64 create a filler Wasm module with `K_MAX_TYPES = 1000000`, matching
  the public fix theme around excessive canonicalized types.
- Lines 66-107 build a Wasm-GC primitive module with 32 i32 fields and exported
  `struct.get` / `struct.set` accessors.
- Lines 120-155 define caged `read64Caged`, `write64Caged`, `addrof`,
  `read64Field`, and `write64Field` using those confused Wasm field accessors.
- Lines 157-159 check that the primitive is alive and abort if it fails.

This is sufficient first-stage evidence for a challenge chain: it provides
in-cage object-field read/write, not full raw virtual-address access by itself.
That matches the local index classification of CVE-2024-6100 as `HS-RW`.

## Second-Stage Evidence

The local second stage has two distinct parts.

First, it converts caged Wasm metadata corruption into raw memory:

- Lines 161-200 build a Wasm module with typed exported functions:
  `do_read(i32)->i64`, `oob_read(i64)->i64`, `do_write(i32,i64)->void`, and
  `oob_write(i64,i64)->void`.
- Lines 207-215 read exported function data and its `func_ref`.
- Lines 217-222 overwrite the `func_ref` fields for `oob_read` and
  `oob_write` with the `func_ref` values from the i32-addressed read/write
  functions. This is exactly the kind of ordinary-heap `WasmFunctionData` /
  `func_ref` coupling that `cf9373a...` moved to trusted space.
- Lines 224-235 derive the cage base and Wasm memory backing store.
- Lines 237-246 define raw virtual-address read/write by passing crafted i64
  offsets through wrappers whose target function now expects i32.
- Lines 248-250 validate the raw-memory primitive by reading the Wasm backing
  store marker. This aligns with `5942a141...`: in-sandbox corruption causing
  an i64 value to reach code expecting i32.

Second, it uses raw memory to redirect a trusted Wasm call target:

- Lines 252-269 compute metadata table, isolate, and trusted pointer table
  addresses using hard-coded d8 layout offsets.
- Lines 271-276 resolve the trusted object for a Wasm function from a
  `WasmFuncRef` trusted-pointer handle.
- Lines 278-299 create and resolve a target Wasm function's trusted internal
  function object.
- Lines 301-328 prepare a small `execvp("/challenge/catflag", argv)` payload in
  writable d8 memory.
- Lines 330-331 overwrite `targetInternal + 0x14` with a native gadget and call
  the Wasm function to reach `execvp`.

This final target is consistent with the Wasm trusted-function hardening around
`WasmInternalFunction::call_target`, exposed `WasmApiFunctionRef`, signature
verification, and trusted pointer table reachability. It is offset-sensitive,
but it is not a sandbox-testing helper path.

## Helper-Free Assessment

The existing exploit is a real chain for this challenge environment.

Reasons:

- The challenge patch only removes dangerous/default d8 global helpers such as
  `read`, `load`, `quit`, etc. It does not add a memory corruption API.
- `common/run` copies a user-supplied script to a temporary file and executes
  `/challenge/d8` as `nobody`. It does not pass `--sandbox-testing`.
- The exploit source does not reference `Sandbox.MemoryView`,
  `Sandbox.getAddressOf`, `%DebugPrint`, `%SystemBreak`, `read`, `load`, or
  other d8 helper APIs.
- The only privileged local helper is `/challenge/catflag`, reached by
  hijacking a trusted Wasm call target to call `execvp`; that is the challenge
  payload, not the memory-corruption primitive.

Limitations:

- It is heavily revision-specific: `D8_BASE`, metadata table, isolate offset,
  trusted pointer table offset, PLT/gadget offsets, and object field offsets
  are all hard-coded.
- I did not complete a fresh harness run because host `pwnshop` failed before
  build/test startup outside the Nix dev shell.
