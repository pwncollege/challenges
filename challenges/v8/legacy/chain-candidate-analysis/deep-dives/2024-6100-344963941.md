## First-Stage Primitive Evidence

Fix-side evidence:

- `422cdc5e...` adds `TypeCanonicalizer::CheckMaxCanonicalIndex()` and calls it
  after canonical type growth.
- The commit message says canonical indices stored in `ValueType` fail when the
  canonical index is too large.
- At `1778ccde...`, `src/wasm/value-type.h` stores heap type indices in a
  20-bit field, and the Wasm module decoder limits only per-module type counts
  to `kV8MaxWasmTypes = 1,000,000`; there is no global canonical-type cap in
  `TypeCanonicalizer`.

Exploit-side evidence:

- `tests_private/exploit.js` creates `fillerModule()` with exactly
  `K_MAX_TYPES = 1000000` types, then creates a second Wasm module using
  struct/ref types.
- The second module exports `readFieldN` and `writeFieldN` functions built from
  Wasm GC struct field get/set opcodes.
- Those exports are used to corrupt the elements pointer of a double array,
  yielding caged `read64Caged`, `write64Caged`, `addrof`, `read32Field`, and
  `write32Field`.
- The exploit contains an explicit first-stage tripwire:
  `prim.readField6([{ marker: 0x1337 }, 13.37, 14.47])`; if it returns zero,
  it throws `"CVE-2024-6100 primitive failed"`.

Conclusion: the private exploit does not just include the CVE-2024-6100 setup;
it depends on it for the object-field read/write primitive that patches later
RegExp internals.

## Second-Stage RegExp / Native-Stack Evidence

Fix-side evidence:

- `8859e5e2...` changes `BacktrackStack::peek()` in
  `src/regexp/regexp-interpreter.cc` from `DCHECK(!data_.empty())` to
  `SBXCHECK(!data_.empty())`.
- The commit message is explicit: prevent reading/popping an empty stack.
- In the vulnerable revision, `BacktrackStack::pop()` calls `peek()`, and
  bytecodes including `POP_BT`, `POP_REGISTER`, `POP_CP`, and `CHECK_GREEDY`
  consume that stack. In release builds, the old `DCHECK` does not protect the
  heap-sandbox boundary.

Exploit-side evidence:

- The exploit constructs a real `RegExp('a'.repeat(128), 'd')`, executes it
  once to materialize internals, then uses the first-stage caged RW to locate
  the `JSRegExp` data object and bytecode array.
- It patches RegExp metadata fields at offsets `0x24` and `0x30`, then replaces
  bytecode words with a hand-built interpreter program.
- The bytecode program uses `BC_SET_SP_TO_REGISTER`, `BC_PUSH_BT`, and
  `BC_POP_REGISTER` to steer the interpreter backtrack stack and produce
  arbitrary native-stack qword writes.
- The emitted writes place `/challenge/catflag`, an argv array, and a small ROP
  chain at hard-coded native addresses. The final chain sets `rdi`, sets `rsi`,
  and calls an `execvp` gadget target.
- `tests_private/test_exploit.sh.j2` runs the JavaScript through
  `/challenge/run` and greps for `pwn.college{...}`, so the existing private
  test expects the full native escape and flag read, not only a crash or caged
  corruption.

Conclusion: the private exploit exercises the b/344963941 class as the second
stage. The first-stage primitive is used to patch RegExp bytecode/metadata; the
RegExp interpreter bug is then used for native-stack writes and command
execution of `/challenge/catflag`.

## Does The Existing Private Exploit Exercise Both Bugs?

Yes, statically verified from the private exploit and challenge harness.

- Without the CVE-2024-6100 Wasm canonical-type confusion, the exported
  struct-field functions should not produce the caged object-field RW; the
  exploit checks this before proceeding.
- Without the b/344963941 RegExp interpreter weakness, the patched bytecode's
  empty/backtrack-stack manipulation should be caught by the new `SBXCHECK`
  hardening rather than becoming native-stack writes.
- The flag path depends on both: Wasm first stage patches RegExp internals,
  RegExp second stage writes the native ROP chain, and the test only passes when
  `/challenge/catflag` prints the flag.

I did not run `pwnshop test` during this pass; the finding above is based on
local source, challenge configuration, and V8 git/diff verification.

## Blockers / Caveats

- Hard-coded native addresses: the exploit assumes stack base
  `0x7fffffffdb90`, d8 load bias `0x555555554000`, gadget offsets, scratch
  addresses, and RegExp object offsets. It is fit for the pinned container, not
  portable across arbitrary builds or ASLR layouts without adjustment.
- Unknown introduction commits: local data proves the pinned revision predates
  both fixes, but it does not identify when either bug was introduced. The
  overlap window should therefore be stated as proven for the challenge revision,
  not as a complete affected-range claim.
- CVE mapping wording: the local index maps `CVE-2024-6100` to likely
  `b/344608204`; the V8 fix commit itself says `Fixed: 344608204` but not the
  CVE number.
- d8-only scope: this is a d8/pwn.college chain with a SUID `catflag` target.
  It should not be described as a complete Chrome renderer escape.
- Fresh dynamic result not recorded here: rerun `pwnshop test
  challenges/v8-exploitation/chain-cve-2024-6100-344963941` before treating the
  challenge as regression-tested after future local edits.
